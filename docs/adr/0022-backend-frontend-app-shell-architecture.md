# App Server, Frontend, App Shells, and App Server Protocol

OpenAIDE will organize product architecture around an App Server, Frontend, App Shells, and a transport-neutral App Server Protocol, using Codex app-server as an architectural reference rather than a naming template. App Server owns product state, workflow decisions, task lifecycle, runtime integrations, capability decisions, and persistence. Frontend renders App Server-owned product state, captures user intent, and keeps only ephemeral presentation state. App Shells are the application forms OpenAIDE runs in: Web App, Desktop App, Mobile App, and VS Code Extension.

OpenAIDE may use similar architectural boundaries to Codex app-server, but it does not adopt Codex's product nouns wholesale. The OpenAIDE user-facing work unit remains Task, not Thread. Turn may be used for a single active Agent run when it matches ACP and product language, and Item may be used inside protocol/rendering models when it is clearer than raw message/update names, but these do not replace Task as the product term.

The App Server Protocol is the bidirectional interface between App Server, Frontend, and App Shells. It carries user intent toward App Server and product state, events, progress, errors, and user-facing requests back toward Frontend or App Shells. The App Server Protocol is transport-neutral: Web, Desktop, and VS Code may use different connection and launch mechanics, but they must not invent separate product protocols or shell-specific App Server APIs.

OpenAIDE will share the Frontend across App Shells as much as possible. Task Page, Task Navigation, Chat, Composer, Settings, product rendering components, and presentation helpers should stay common. App Shell-specific UI is limited to shell chrome, placement, menus, routing adapters, window behavior, and capability affordances injected through narrow composition points.

App Shells call App Server through the same App Server Protocol product API. Shell-specific code may handle native commands, auth/session entry points, process connection, file pickers, menu actions, or window routing around that API. App Shells must not own product workflow decisions such as task lifecycle transitions, Agent Status interpretation, stale response handling, settings truth, or persistence and reload behavior.

App Server owns Agent discovery, setup validation, auth/setup status, and Agent Status interpretation. App Shells may provide environment inputs such as workspace location, process environment hints, or user-selected executable paths, but they do not decide whether an Agent is connected, setup required, auth required, failed, or unsupported.

Agent Settings are App Server-owned product state. App Server owns built-in and Custom Agent definitions, enabled state, launch-affecting identity changes, status, and validation. App Shells may provide native helpers such as secret storage, file pickers, and environment hints, but shell-local configuration UI must not become the source of truth for Agent definitions. Frontend renders Agent Settings and sends edit intents through App Server Protocol.

Secret values are App Shell-owned capability storage, while App Server-owned Agent settings store only secret references or variable names. When launching or authenticating an Agent, App Server requests required secret material through the App Shell capability provider instead of persisting secret values in App Server state, Frontend state, logs, or Chat history.

In v1, a usable Agent must support ACP session listing and at least one history-open path. Agents must advertise `sessionCapabilities.list` and either `loadSession` or `sessionCapabilities.resume`. `loadSession` allows OpenAIDE to hydrate Chat by calling `session/load`; `sessionCapabilities.resume` allows OpenAIDE to reconnect while using locally persisted normalized Chat history. Agents without these capabilities are unsupported for OpenAIDE's task history model until a deliberate fallback is designed.

Agent identity is an App Server-owned stable Agent definition id. Built-in Agents have fixed ids; Custom Agents receive generated stable ids at creation. Display names and executable paths are not identity. Renaming a Custom Agent preserves Agent Identity and updates labels for existing projected Tasks. If a user edits launch-related settings, or OpenAIDE changes a Built-in Agent launch definition without an explicit migration path, App Server treats the result as a new Agent identity and deletes/cleans up local overlays and normalized Chat history for the old identity. Agent Identity cleanup is local-only by default and must not call ACP `session/delete`; destructive deletion of Agent-owned sessions remains a separate explicit feature.

Launch-related settings include command, args, environment values or names that affect process behavior, working mode flags, Agent binary source or version channel, built-in fallback launch policy, and auth profile/account selection that changes which underlying Agent session store is used. Before applying launch-related edits, OpenAIDE must warn the user that the edit creates a new Agent identity and cleans up old OpenAIDE local overlays and normalized Chat history for the previous identity. Non-launch metadata, such as display name, icon, color, description, or enabled state, does not change Agent Identity.

Disabling an Agent keeps the same Agent Identity but removes that Agent from launch, session listing, and normal history projection while disabled. Re-enabling the Agent reconnects and lists sessions again under the same Agent Identity. Local normalized Chat history may remain while an Agent is merely disabled; identity cleanup or explicit deletion removes it.

Enabled Agents that are disconnected, failed, auth-required, setup-required, or unsupported remain visible in Settings with Agent Status. Their sessions are omitted from normal history until the Agent connects and lists successfully. Global history should indicate when some enabled Agents are unavailable so users understand the visible history may be incomplete and can navigate to Settings for recovery.

App Shells discover and supply workspace or project context. App Server owns how that context affects Tasks, Agents, persisted state, MCP resolution, and permission decisions. Frontend displays workspace context and asks users to choose or change it through App Server Protocol rather than reading shell-specific workspace APIs directly.

Projects are lightweight OpenAIDE records for user work areas, such as folders, workspaces, or repositories. Project identity is based on canonical local path. A Project should have a stable id derived from or mapped from that path, display name, path metadata, and recency metadata. App Server owns Project records and canonicalization. App Shells report candidate project facts, such as selected folder path, workspace name, or open-folder events; App Server matches or creates the Project record by canonical path and returns canonical Project Context to Frontend.

If a folder is moved or renamed, it is treated as a different Project because its canonical path changed. OpenAIDE may later add an explicit user-controlled migration or linking tool, but v1 must not infer moved-project identity automatically.

Tasks are global within an OpenAIDE state root and are always associated with a Project. A Task's project context is fixed at creation time; moving work to a different project context requires creating, continuing, or forking into another Task rather than mutating the existing Task in place. This matches ACP session semantics, where `cwd` is chosen at session creation, load, or resume boundaries and is not a first-class mutable property of an active session. Web App and Desktop App should expose global task history grouped and filterable by project context. VS Code Extension defaults to the current workspace context because the user opened that shell inside an IDE workspace, but this is a default shell view/filter rather than the storage owner of the Task.

When Web App or Desktop App creates a Task, the user selects the workspace or project path as part of the new Task flow. The selected path becomes the Task's Project Context through App Server project canonicalization. VS Code Extension defaults new Tasks to the current workspace context.

Web App and Desktop App use project-first navigation because they expose global task history across project contexts. VS Code Extension uses task-first navigation within the current workspace context because the IDE already provides the project frame. Shared Frontend must support both compositions from common product data and components rather than forking Task, Project, Chat, or Settings behavior by shell.

In Web App and Desktop App, history belongs in the sidebar and the main initial surface should emphasize starting a new Task. Opening a new Task page includes choosing the workspace or project path and Agent before App Server creates the Task and associated Project Context. OpenAIDE should not create draft Tasks without that required start context. After the required start context is selected, App Server creates the Task and immediately starts the Agent Native Session with ACP `session/new`. The first user prompt is sent later, only when the user sends it from the Task Page composer.

Web App and Desktop App global history merges sessions from all connected Agents that satisfy OpenAIDE's v1 session-list requirements. VS Code Extension also merges sessions from all connected/list-capable Agents, filtered to the current workspace context by default. Task rows must expose Agent identity so users can understand which Agent owns each session. The app-wide default Agent affects new Task creation, not which existing Tasks are visible.

If `session/new` fails while opening a new Task page, OpenAIDE shows the error as renderable Task preparation state with explicit retry, change Agent, discard, or setup actions. Task creation is immediate and durable after the required Project Context and Agent are selected, but native Agent setup is asynchronous so the UI can render and remain responsive while ACP setup runs.

Task creation is split between core product rules and app-server ACP I/O. `openaide_core::task_creation` validates the product rules and prepares a creation plan: Project exists or is canonicalized, Agent is selected and connected, no draft Task is created, and a Task record can be committed before unbounded Agent I/O. The app-server `task/create` handler commits the Task with preparing state, asks `agent_runtime` to perform ACP `session/new`, and streams ordered Task snapshot updates as native session setup, Configuration Options, slash commands, authentication, or failures become known. `state_stream` may emit task-list updates as soon as the durable Task exists and must emit later readiness or failure state changes after persistence.

On success, `task/create` returns a committed `TaskSnapshot`, not only identifiers. The snapshot includes the OpenAIDE Task id, Agent Identity, Project Context, Task revision, empty renderable Chat state, preparation state, Agent Configuration Option state, slash command state, send capability state, and pending requests, usually empty. Native Session id is App Server-owned and appears in protocol only through safe render state when needed. This lets Frontend navigate directly to the Task Page without racing a subscription event; the client can still subscribe immediately afterward for updates. If synchronous validation fails before Task commit, `task/create` returns a protocol error. If Agent setup fails after commit, the Task remains renderable with preparation failure state and user-visible recovery actions.

Composer state stays in Frontend until `task/send`. Draft text, visible attachment rows, row order, invalid selection rows, and unresolved candidate rows are ephemeral presentation state. App Server does not own a hidden server-side composer session and does not persist drafts or row lists. App Server owns only the server-backed records and decisions that Frontend references from the composer: pre-send attachment handles, non-sendable attachment candidates, capability state, validation responses, candidate confirmation or consumption into handles, and final `task/send` validation.

Unsent composer state is scoped to the Frontend client that owns it. If another client sends a message in the same Task, this client receives the normal Task snapshot or event showing the Task as running, but keeps its draft text, attachment rows, row order, invalid rows, and unresolved candidate rows as local editable state. Frontend disables Send until the active turn finishes, because App Server allows only one active turn for the Task, but it does not clear, merge, synchronize, or auto-send the draft. When the user later sends, Frontend creates a fresh send attempt and App Server revalidates attachments, options, Agent capability, and Task state against the current Task context. If a stale client submits while the Task is already running, App Server returns a structured Task-running error and commits no user message or turn.

Before re-enabling Send after another client's active turn finishes, Frontend performs a lightweight App Server refresh for the local unsent composer rows. The refresh updates App Server-provided attachment capability and validation state in place, can mark rows stale, invalid, or non-sendable, and still leaves authoritative validation to `task/send`. It follows the same content boundaries as manual refresh: it must not read `fileReference` contents, capture `embeddedSnapshot` bytes, silently replace handles, consume candidates, confirm embedding, create sent metadata, or reorder rows. This is a Task-state-triggered validation pass for an existing local draft, not filesystem watching and not draft synchronization.

If that post-turn idle refresh fails because of transport loss, storage error, App Server error, or another non-validation failure, Frontend keeps Send disabled and preserves draft text, row order, row ids, and row validation state unchanged. It renders a recoverable composer validation error with a manual retry action that runs the refresh again. A failed refresh is not treated as proof that the visible attachments or options are sendable against the current Task state, and Frontend must not proceed to `task/send` until the refresh succeeds or the user changes the composer in a way that obtains fresh App Server validation.

Local composer blockers determine whether Frontend may enable Send for an idle, non-submitting composer. Blockers include an empty or whitespace-only draft unless App Server capability state says the current Agent and Task can accept an attachment-only user message and at least one visible attachment is sendable, unresolved attachment candidates, invalid rows, stale rows, needs-refresh rows, a failed validation refresh, Task running state, composer submitting state, missing required Agent or run options, and any App Server-reported capability or Task state that makes send unavailable. Attachment-only sends are allowed only when the App Server-provided capability state allows them, the visible attachments are sendable, and no other blocker exists. Otherwise empty text remains a blocker even if attachments are present. Frontend may compute the button state locally, but only from local draft emptiness plus App Server-provided row validation, capability, option, and Task state. It must not infer sendability from local paths, file extensions, shell kind, Agent name, or stale cached assumptions.

If attachment-only send is not allowed but the visible attachments are otherwise sendable and draft text is empty or whitespace-only, Frontend renders a composer-level validation message such as "Add a message for this Agent." It keeps attachment rows in their current valid/sendable state and must not mark them invalid, stale, or failed, because the blocker is the Agent/Task message-shape capability rather than file validation. If a stale or buggy client submits this shape anyway, App Server returns a structured message-shape validation error, commits no user message or turn, and leaves attachment handles under the normal failed-validation lifetime rules.

User messages use a part-based shape in protocol and storage rather than a plain prompt string. Message parts include text and may include attachment references when the selected Agent adapter and App Server capability state support them. OpenAIDE v1 composer and `task/send` support message-level attachments only: at most one text body plus an ordered attachment list. App Server builds the sent message with non-empty text first when present and attachments afterward in user-selected order. When attachment-only send is allowed and the draft text is empty or whitespace-only, `task/send` normalizes the text part to absent or empty according to the protocol shape and must not persist or deliver whitespace as prompt text. The committed message body is the ordered attachment list. Inline attachment placement inside text is deferred, but `task/send`, normalized Chat, request fingerprints, snapshots, and generated bindings must preserve the part model so file references, images, inline placement, or other future inputs do not require rewriting core storage.

Message-level attachment order is user-controllable and semantically durable. The composer preserves selection order and may allow users to reorder attachments before send. `task/send` commits the submitted order, normalized Chat renders that order, request fingerprints include that order, and Agent delivery follows that order. Reordering attachments before send changes the request fingerprint; reordering after send is not a Chat edit in v1 and requires a new send.

Attachment references are App Server-owned metadata handles, not raw file content in the App Server Protocol. A reference may expose an opaque attachment id, kind, safe display label, size or media type when known, and capability or availability state. Raw local paths, path-derived identifiers, file contents, and shell-specific picker details stay server-side and are not included in protocol snapshots. App Server resolves the reference when delivering a turn to an Agent adapter that supports that attachment kind; unsupported references fail validation before `task/send` commits a turn.

Local file message parts distinguish `fileReference` from `embeddedSnapshot`. `fileReference` is the default for workspace files: it records a server-side resolver to a local file and lets the Agent read current content through its normal filesystem or resource-link path when capabilities allow it. `embeddedSnapshot` is explicit, size-capped, and means OpenAIDE intentionally captures file contents as prompt context for that send. UI must not collapse both modes into one vague attachment concept because their privacy, staleness, retry, and Agent-delivery behavior differ.

Attachment mode is chosen per file, not as one global composer mode. When a valid local file is selected, App Server defaults it to `fileReference` when the current Task, Native Session, and Agent adapter support that mode. Frontend may let the user switch an individual file to `embeddedSnapshot` when App Server capability state says it is available. Mode changes revalidate that handle, can move it into stale or invalid state, and change the `task/send` request fingerprint.

If App Server emits changed Agent or Task attachment capability state while a Frontend client has a local draft, Frontend updates the rendered capability state without rewriting the user's attachment intent. It preserves draft text, row order, selected attachment modes, and existing row identities. Affected rows may become needs-refresh, invalid, or non-sendable, and Send may be disabled until explicit refresh or user action resolves them. Frontend must not auto-switch `fileReference` to `embeddedSnapshot`, auto-confirm embedding, drop attachments, reorder rows, or auto-send a text-only prompt. If the user wants to switch modes after the capability change, that is an explicit mode change or confirmation path and gets fresh App Server validation; `task/send` still revalidates authoritatively before commit.

When a capability change marks local composer rows needs-refresh and the Task is idle, Frontend automatically runs a lightweight refresh for the affected rows only, provided the composer is not submitting. The refresh uses the same boundaries as manual refresh: it must not read `fileReference` contents, capture `embeddedSnapshot` bytes, switch attachment modes, reorder rows, consume candidates, confirm embedding, create sent metadata, or replace handles silently. If the Task is running, Frontend defers to the post-turn idle refresh path. If the capability-change refresh fails for transport, storage, or App Server reasons, Frontend keeps Send disabled, preserves rows unchanged, and shows the same manual retry recovery as other failed validation refreshes.

If the automatic capability-change refresh succeeds, Frontend applies the returned validation state in place and immediately re-enables Send when the Task is idle and all local composer blockers are cleared. It must not require another user action only because the successful refresh was automatic. A successful refresh still preserves selected attachment modes, row identities, and row order; it never auto-confirms embedding, switches modes, drops rows, or sends the prompt.

If `fileReference` is unavailable but `embeddedSnapshot` is available, App Server must not auto-default to embedding file contents. The selection flow must require explicit per-file user choice, or a clear per-file mode switch, before a sendable embedded handle exists. This keeps content embedding opt-in because it has different privacy, size, retention, and Agent-delivery behavior from a filesystem reference.

When only `embeddedSnapshot` is available for a selected file, App Server may return a non-sendable attachment candidate instead of a pre-send handle. The candidate is backed by App Server validation state and contains only protocol-safe metadata, so Frontend can show an "embedding required" row and let the user confirm per file without browsing again. Candidates are scoped and ephemeral like pre-send handles, but they cannot be passed to `task/send` and do not count as sendable attachments. After explicit user confirmation, App Server revalidates and consumes the candidate into a new embedded pre-send handle id or returns structured validation errors.

Embedded candidate confirmation is batch-oriented. `attachment/confirmEmbedded` accepts candidate ids, revalidates each candidate under the current Task, Native Session, Agent capability, allowed-root, type, and size rules, and returns per-id outcomes. Successful outcomes consume the candidate and return a new sendable embedded pre-send handle id; candidate ids remain non-sendable by type and are never mutated into sendable handles. Failed outcomes include structured validation errors; those candidates remain non-sendable or become invalid rows in Frontend until the user removes them, fixes the issue, or selects again. Partial success is allowed, but visible failed candidates still block Send until resolved.

`attachment/confirmEmbedded` does not capture file bytes. It records explicit user confirmation and revalidates enough to create a sendable embedded pre-send handle. Actual `embeddedSnapshot` bytes are still read, decoded, size-checked, hashed, and captured only during `task/send`. If the file changes after confirmation, the send-time contents are what get embedded. If the file becomes invalid after confirmation, `task/send` fails validation without committing the user message or turn.

Embedded size limits are enforced at confirmation for early feedback and at `task/send` as the authority. `attachment/confirmEmbedded` may reject files that are already too large based on current file size, and may return aggregate warnings or validation state based on the current set of embedded candidates and handles. `task/send` still rechecks the exact bytes and aggregate embedded snapshot total immediately before commit. If the exact sent bytes exceed per-file or aggregate limits at send time, App Server commits no user message or turn.

Send-time aggregate embedded size counts only confirmed embedded handles because unresolved candidates are not sendable and cannot be part of the committed request. For UI feedback, App Server may return projected aggregate warnings or validation state that includes unresolved candidates, but those projections must be labeled as projected and non-authoritative. The authoritative aggregate is computed from the exact bytes captured for submitted embedded handles during `task/send`.

Visible unresolved attachment candidates or invalid attachment rows block sending. Frontend must disable Send or surface a blocking validation state until the user confirms a candidate into a handle, removes a candidate, fixes an invalid row, or removes the invalid row. Frontend must not silently omit visible attachment rows when building `task/send`, because users would reasonably believe every visible file was included. If a client submits candidate ids or stale/invalid handles anyway, App Server returns structured validation errors and commits no user message or turn. App Server still revalidates submitted sendable handles as the source of truth.

`fileReference` resolver data may store raw local paths only in App Server-owned local storage. Protocol snapshots expose safe display labels, project-relative labels when possible, opaque attachment ids, kind, size or media type when known, and capability or availability state, but not raw paths. Support Export follows the path tokenization/redaction policy rather than exposing raw resolver data. Before `task/send` commits a turn, App Server revalidates each `fileReference` against the Task's canonical Project Context and the active additional roots that are allowed for that Native Session. A path that moved, no longer exists, is no longer under the Task Project Context, or is outside the active allowed additional roots fails validation before the user message and turn are committed.

`fileReference` request fingerprints do not include file contents or file content hashes. They include attachment mode, message-level order, stable App Server resolver identity, canonical validation facts, and safe metadata needed to detect changed send intent. A `fileReference` is live/current by design; hashing content would make it behave like `embeddedSnapshot`, force unnecessary file reads, and create confusing stale-content conflict behavior.

`fileReference` send-time validation must not read file contents. It validates canonical path identity, existence, regular-file type, blocked product-internal paths, allowed roots, obvious OS permission or access failures, basic accessibility needed to produce a usable Agent reference, and current capability state. It must not sniff full MIME type by reading bytes or hash contents. If richer content inspection is needed later, it belongs to an explicit `embeddedSnapshot` or future content-inspection feature, not reference validation.

`fileReference` validation requires enough non-content local access for App Server to stat and canonicalize the entry, reject obvious permission or access failures, detect regular-file type, block product internals, evaluate allowed roots, and produce an Agent-usable reference. It does not require OpenAIDE to read file contents. If App Server cannot stat or canonicalize the entry, cannot perform the non-content access checks needed for Agent delivery, or can determine the selected local Agent will not be able to access the path through the Task Project Context or allowed additional roots, selection or `task/send` fails with a structured validation error. If validation succeeds and the Agent later cannot read the delivered reference for Agent, tool, or runtime reasons, OpenAIDE renders that as Agent/tool activity or turn failure rather than attachment delivery failure.

Additional roots are per Task and Native Session context, not app-wide settings. App Server stores the Task's allowed root set alongside Task launch context and uses it for file reference validation, session load/resume planning, and UI capability state. For ACP adapters, App Server sends the full intended `additionalDirectories` list on `session/new`, `session/load`, and `session/resume` only when the initialized Agent advertises `sessionCapabilities.additionalDirectories`; otherwise only the Task Project Context is active. Because ACP does not implicitly restore old additional roots on load or resume, OpenAIDE must treat the stored Task allowed root set as the source for every lifecycle request.

Additional roots are immutable after Task creation in v1. Changing filesystem scope requires creating, continuing, or forking into a new Task with a new launch context and, when applicable, a new Native Session lifecycle request. App Server must not silently expand the allowed root set of a running Native Session because that changes the filesystem scope available to the Agent and invalidates earlier file-reference validation assumptions.

Allowed-root labels are App Server-generated in v1 and are not user-editable. App Server derives them from safe product context such as project name, root role, safe directory name, and generated disambiguators. Labels must never expose raw absolute paths. Editable root display names are deferred because they introduce persistence, collision, migration, and privacy policy decisions that are separate from attachment delivery.

Generated allowed-root labels are stable Task-scope metadata. App Server stores safe generated labels with the Task's allowed-root records when the Task is created, continued, or forked. Reload, history rebuild, and Chat rendering reuse the stored labels instead of recomputing them from current filesystem or project state. New Task scopes may receive newly generated labels. This keeps old attachment labels auditable after project rename, directory rename, or later disambiguation changes.

A single composer message may include ordered attachments from multiple roots that are already allowed for the Task and Native Session. Each attachment validates against its own root, and App Server preserves the user's cross-root attachment order in `task/send`, Chat rendering, delivery, and request fingerprints. Safe labels should include enough root context to disambiguate same-label or same-relative-path files from different allowed roots without exposing absolute paths. Files outside the current allowed root set still require an explicit new, continue, or fork scope-change flow.

When allowed roots overlap, App Server preserves the root the user selected in the App Server-backed file browser for safe labels and relative delivery metadata. For shell-native selections where no browser root was selected, App Server chooses the most-specific allowed root containing the selected file. This attribution affects safe labels and relative metadata only; it does not relax allowed-root validation or duplicate detection.

If a user attaches a `fileReference` outside the Task's allowed roots, App Server returns a structured validation error and `task/send` does not commit the user message or create a turn. For an existing Task, Frontend may offer to start a new Task, continue, or fork into a new Task with that root explicitly included in the new launch context. In the new Task flow, Frontend may let the user explicitly add the root before Task creation. OpenAIDE must not auto-add additional roots from attachments because that silently expands the Agent's filesystem scope.

`embeddedSnapshot` sources must pass the same Task allowed-root validation as `fileReference`. Even though embedding file contents does not grant ongoing filesystem access, it still sends file contents into the Agent context. OpenAIDE v1 must not embed contents from outside the Task Project Context or active allowed additional roots. If the user needs an outside file, they add that root explicitly in a new Task, continue, or fork launch context before sending.

`embeddedSnapshot` bytes are captured at `task/send`, not at file selection. The pre-send attachment handle may carry preview metadata, but App Server reads the file, revalidates allowed roots and type/size limits, hashes the exact bytes, and embeds the current file contents during the send commit. If the file changes after selection but before send, the sent snapshot reflects the current bytes at send time and durable metadata records that hash and size. If the file no longer exists or no longer validates, `task/send` fails validation without consuming the handle.

`embeddedSnapshot` v1 accepts UTF-8 text only. Non-UTF-8 or otherwise undecodable files are rejected with structured validation errors. OpenAIDE does not normalize line endings; the decoded text delivered to the Agent preserves the file's original line endings. A UTF-8 BOM may be stripped for display or Agent delivery, but the local content fingerprint is computed from the raw captured file bytes before decode, BOM stripping, display cleanup, or delivery formatting. If BOM stripping occurs, durable safe metadata records that delivery cleanup fact without storing file contents.

When an `embeddedSnapshot` is delivered to an Agent, the Agent should receive the text content plus safe resource metadata when the adapter supports metadata fields. Safe delivery metadata may include `sentAttachmentId`, safe display label, text or media type, byte size, and a synthetic non-file resource id or URI. It must not include raw local paths, file URIs, path-derived ids, shell picker metadata, or resolver ids. This gives the Agent context for the embedded text without leaking filesystem identity.

If an Agent adapter cannot carry resource metadata for `embeddedSnapshot`, App Server must not inject synthetic headers, labels, delimiters, comments, or file names into the embedded text. Metadata remains metadata, represented locally and in safe delivery bookkeeping. The delivered embedded text stays the decoded file content under the UTF-8, line-ending, and BOM rules above. Injecting labels would mutate the user-provided content, make Agent-visible text diverge from the captured bytes in surprising ways, and create confusing replay, hash, and diff behavior.

Missing adapter support for embedded resource metadata does not block an explicit `embeddedSnapshot` send in v1. App Server capability state may expose a subtle metadata-unavailable indicator so Frontend can set expectations, but it must not turn into a blocking warning or confirmation loop after the user has explicitly chosen embedding. Delivery status tracks content delivery in v1. OpenAIDE does not separately report metadata delivery success or failure unless a later feature designs that status model.

Synthetic embedded resource ids or URIs are stable only within the committed sent attachment. App Server derives them from `sentAttachmentId` or another opaque App Server id so idempotent retries for the same `task/send` produce the same Agent-facing resource identity. Different sends get different synthetic ids even when they reference the same file, same display label, or same file contents. Synthetic embedded resource identity must not be path-derived, content-derived, reusable across sends, or usable for filesystem access.

Pre-send and confirmation preview metadata may include safe display label, media or text type, current size when known, and App Server validation state, but not content hashes. The durable embedded content fingerprint belongs to the exact bytes captured at `task/send`; an earlier preview hash would look authoritative while becoming stale if the file changes before send.

OpenAIDE v1 does not warn merely because an attached file changed after selection. `fileReference` is live/current by definition, and `embeddedSnapshot` is defined as send-time capture. App Server may refresh preview metadata opportunistically, but final send-time validation and durable metadata are authoritative. Selection-time stability warnings can be added later only if user workflows prove they need that distinction.

Elapsed time alone does not create a composer warning. A long-waiting live composer row keeps its last known validation state and must not show a generic stale or validation-may-be-stale warning only because time passed, a keepalive ran, or another turn delayed Send. Frontend may show stale, invalid, needs-refresh, or unknown states only from App Server validation results, Task or capability state changes that require refresh, or a failed refresh. `task/send` remains the authoritative validation point before committing the user message.

OpenAIDE v1 does not use filesystem watchers to proactively refresh composer attachment validation state. App Server revalidates only on defined validation passes: selection, mode change, embedded confirmation, live open/reveal, manual refresh, capability-change refresh for affected local draft rows, post-turn idle refresh for a local draft, and `task/send`. Watchers are deferred because they add OS-specific behavior, battery and performance cost, race noise, and confusing background UI changes. A stale file discovered by one of these validation passes still updates composer validation state through that response.

The composer may expose a subtle manual refresh attachments action. It batch-revalidates all live pre-send handles and non-sendable attachment candidates in the current composer, returns per-row validation state, and updates Frontend rows without committing a user message or creating a turn. Manual refresh follows the normal mode-specific validation boundaries: it must not read `fileReference` contents, must not capture `embeddedSnapshot` bytes, and must not turn an unresolved candidate into a sendable handle without the explicit confirmation path.

Manual refresh preserves composer row order and existing ids for still-valid pre-send handles and non-sendable candidates. App Server returns outcomes keyed by the submitted handle or candidate ids, and Frontend updates metadata and validation state in place. Refresh must not silently replace handles, consume candidates, confirm embedding, create sent attachment metadata, or reorder rows. If validation fails, the existing row becomes stale, invalid, or non-sendable until the user removes it, fixes the issue, confirms through the explicit confirmation path, or reselects the file.

While `task/send` is in flight for a composer, Frontend treats the composer as submitting and locks all send-affecting controls for that composer. The user must not edit draft text, reorder attachments, add or remove rows, release handles, change attachment modes, confirm candidates, change Agent or run options, manually refresh attachments, live open/reveal attachments, or trigger another send until the request succeeds or fails. If send fails validation, App Server leaves unconsumed handles usable under the existing failure rules and Frontend exits submitting state with the same draft, same row order, same still-live row ids, and updated validation errors. This avoids races with handle reservation, consumption, request fingerprints, and validation state updates.

After successful `task/send`, Frontend clears the submitted composer immediately when App Server returns the durable `turnId` and updated Task snapshot. It does not wait for the first Agent output event because the user message is already durable and visible through Chat. Keeping submitted text or rows in the composer after durable acceptance would invite duplicate sends and make consumed handles look reusable. Agent output streams later through normal Task events while the composer is ready for the next user message.

If the successful `task/send` response is lost after App Server committed the send, Frontend stays in submitting recovery state and retries `task/send` with the same idempotency key and request fingerprint. If App Server returns the existing `turnId` and current Task snapshot, Frontend treats that as the original success and clears the composer. If App Server returns `idempotencyKeyConflict` or validation failure, Frontend exits submitting state and restores editable draft and rows with the server error. Frontend must not switch to a new idempotency key for recovery because that can create duplicate sends.

Submitting recovery state is same-client ephemeral shell state, not App Server product state and not shared across clients. Before issuing `task/send`, Frontend may write a recovery record containing the idempotency key, submitted request data needed to retry the same protocol input, draft and row render state, and `clientInstanceId` to App Shell-managed session state. On browser reload, VS Code webview recreation, Desktop remount, or Mobile remount, a client with the same `clientInstanceId` restores only a locked submitting UI, loads or subscribes the Task snapshot first, and clears the record if the accepted send is already visible. If the accepted send is not visible, it retries `task/send` with the same key and submitted request data so App Server can match the canonical request fingerprint. If the recovery record is missing, storage was unavailable, or the client identity changed, Frontend must not auto-resend; it relies on Task snapshot and Chat state. App Server remains the source of truth for the canonical fingerprint, turn existence, and validation outcome.

If the same-key recovery retry reaches App Server and App Server reports that no turn was committed because the original attachment handles are expired, released, invalid, or no longer authorized, Frontend exits submitting state into an editable composer. It restores the submitted prompt text and submitted row order from the recovery record, but renders the affected attachment rows as stale non-sendable rows with the server's structured errors. The user must remove, reselect, or reconfirm those attachments under current validation rules before sending again. Frontend must not silently omit those rows from the next send, drop them without explanation, or auto-send a text-only prompt, because the user's visible send intent included the attachments.

`embeddedSnapshot` contents are outbound send-time material, not normalized Chat history in v1. OpenAIDE persists safe metadata such as attachment id, display label, kind, size or media type, a local content fingerprint, and a flag that contents were embedded at send time. It does not persist full file contents in normalized Chat, Task snapshots, support exports, or local diagnostics unless a later explicit retention feature is designed with user warning, size limits, and export redaction behavior. Reloaded Chat can show that a snapshot was sent, but not reconstruct or display the captured file body from OpenAIDE storage alone.

The `embeddedSnapshot` content fingerprint is a local cryptographic hash of the raw file bytes captured at `task/send`, before any decode, BOM stripping, display cleanup, or delivery formatting. App Server can use the full local hash for idempotency fingerprints, internal dedupe of send material, and diagnostics that stay on the machine. UI, protocol snapshots, support exports, and normal diagnostics may expose only redacted or truncated fingerprints when useful for correlation. Fingerprints must never be derived from local paths, filenames, display labels, or Agent-provided ids.

`embeddedSnapshot` v1 supports text files only. Default limits are 256 KiB per file and 1 MiB total embedded snapshot content per send. These limits are checked during confirmation for early feedback and rechecked authoritatively at `task/send`. Filesystem-backed binary and audio embedded snapshots remain disabled even when an external protocol such as ACP has capability-gated shapes for them, until OpenAIDE designs explicit UI, validation, size policy, rendering, storage metadata, and adapter mapping. Image upload and paste are desired v1 features, but they use a separate App Server-owned upload attachment flow with explicit validation, preview metadata, storage policy, and Agent capability mapping; they are not raw local-path `embeddedSnapshot` attachments. Files that exceed the cap, are unsupported, or cannot be decoded as text are either sent as `fileReference` when valid for the Task and Agent, handled by an upload attachment flow when supported, or rejected with a structured validation error before `task/send` commits a turn.

Frontend renders attachment controls from App Server-provided Task attachment capability state. That state includes whether the current Task and Agent can use `fileReference`, whether `embeddedSnapshot` is available, whether attachment-only sends are allowed, text snapshot limits, per-file validation status, aggregate validation errors, and recovery actions such as starting or forking a Task with an added root. Frontend does not infer product decisions from file extensions, MIME guesses, paths, shell kind, or Agent names. App Shells provide shell-specific file picker, open-file, or reveal-file affordances, but App Server owns attachment capability, validation, and send eligibility.

App Shells hand selected local files or uploaded file bytes to App Server through attachment protocol methods. Desktop and VS Code may use native file pickers or editor selection APIs to collect shell-private local file facts, then pass those facts to App Server for canonicalization and validation. Web App may use browser upload or paste to collect user-selected file bytes, then pass those bytes to App Server for upload validation and handle creation. App Server returns protocol-safe attachment handles or non-sendable attachment candidates for Frontend to render. Raw local paths may exist in shell-private request inputs and App Server resolver storage, but they must not become Frontend state, Task snapshots, support export fields, or normal diagnostics. Web App v1 uses an App Server-backed file browser rooted in the Task's allowed roots for local file references, and explicit browser upload/paste for uploaded files such as images. Browser paths, file blobs, and client-only metadata are never the source of product attachment truth after selection; App Server-owned handles, validation state, safe labels, and send-time checks are authoritative.

Hard-invalid attachment selections create no usable pre-send handle or confirmation candidate. If a selected file is outside allowed roots, blocked as OpenAIDE internals, missing, unreadable, too large, or unsupported for the selected attachment mode, App Server returns structured selection or validation errors and leaves existing valid handles intact. Frontend may render those errors as composer rows or notices, but those rows are presentation state only and cannot be included in `task/send`.

Multi-file attachment selection is partial-success. In one selection response, App Server returns pre-send handles for valid files and structured errors for invalid files. Existing handles remain intact, and newly returned valid handles preserve their relative selection order. Frontend renders valid attachments plus error rows or notices, but `task/send` only accepts valid handle ids; invalid rows are not sendable attachment records. This partial-success rule applies to selection only. A later `task/send` remains atomic: if any submitted handle is stale or invalid at send time, App Server commits no user message or turn.

Duplicate attachment selection is explicit validation, not silent dedupe. If the user selects the same canonical file again in the same composer, App Server returns a structured `duplicateAttachment` validation error and creates no duplicate handle. This applies even when the requested modes differ, such as `fileReference` plus `embeddedSnapshot` for the same file; v1 treats canonical file identity as duplicate regardless of mode. In a multi-file selection, non-duplicate valid files still receive handles under the partial-success rule, and duplicate rows remain Frontend presentation state only. Duplicate detection uses App Server canonical file identity, not display labels or path strings supplied by Frontend.

Previously valid pre-send attachment handles may become stale. A handle can move into a stale or unavailable validation state if the file is deleted, permissions change, size or type changes, the selected attachment mode is no longer supported, Agent capability state changes, or the Task/Native Session context changes. Stale handles may stay visible in the composer with App Server-provided validation state, but `task/send` revalidates every handle and blocks commit until the user removes the stale item or reselects a valid file. Pre-send handles never migrate across new, continued, or forked Tasks; if a new Task context changes allowed roots or capabilities, attachments must be selected again and App Server creates fresh handles for that Task.

The App Server-backed file browser exposes only entries scoped to the Task's allowed roots. Root and entry identities are opaque protocol ids such as `rootId` and `entryId`; Frontend receives safe display labels, project-relative display paths when possible, entry type, size when known, and basic availability or validation state. Directory listing and file selection are App Server methods, not browser filesystem APIs, so symlink policy, hidden-file policy, permissions errors, stale entries, and allowed-root validation stay centralized. Listing, selection, snapshots, and support export must not expose absolute paths.

V1 file browsing supports direct directory listing only. It does not include recursive search, background filesystem indexing, or cross-root search. Frontend may filter already loaded entries for presentation, but that filtering is non-authoritative and must not become a validation source of truth. Recursive search is deferred because it needs separate privacy, performance, stale-result, hidden-file, ignored-file, product-internal denylist, and cancellation semantics.

Directory listing is paginated in v1. App Server owns page size limits, stable sorting for a listing request, cursor issuance, refresh, cancellation, and per-page validation. Frontend renders the rows it has loaded and requests more; it must not assume that the first response contains the whole directory. This prevents large directories from blocking UI rendering, overloading transports, or turning filesystem latency into awkward request timeouts.

The v1 default directory sort is App Server-owned and deterministic: directories first, then files, each ordered by case-insensitive natural display-label sort with a stable tie-breaker. Frontend may visually mark rows or filter already loaded rows for presentation, but it must not authoritatively reorder paginated results because that breaks cursor semantics. Later sort variants can be added as explicit listing parameters that App Server applies before issuing cursors.

Directory listing cursors are opaque, ephemeral App Server resources scoped to `clientInstanceId`, root, directory, sort, and listing generation. They are not durable product state and must not survive browser reload, webview recreation, Task Page remount, or explicit refresh. Reload or refresh starts a new listing. If filesystem changes or App Server cleanup invalidate a cursor, the next page request returns a structured stale-cursor error and Frontend refreshes the directory instead of trying to repair the old cursor.

File browser entry ids are opaque handles valid only within one listing generation and client scope. They are inputs to App Server selection methods, not durable filesystem identities, protocol snapshot identifiers, or attachment ids. Refreshing a directory creates new entry ids. Already selected attachments keep their separate pre-send handles and are revalidated independently, so refreshing the browser must not invalidate a valid composer row by itself.

File browser listings return unavailable rows for entries whose safe name and basic type can be observed but cannot be opened or attached because of permissions or access. Those rows may include a safe label, entry type when known, availability state, and structured reason, but not raw paths or resolver details. Entries that the OS does not reveal are omitted. If the user tries to open or select an unavailable row, App Server returns a structured validation error rather than creating a pre-send handle or exposing a path.

File browser rows and sent attachment metadata include root context when needed to distinguish entries from different allowed roots. That context must use safe root labels and relative segments only; it must not expose absolute paths, raw root paths, or shell picker details. The goal is to make cross-root attachments understandable in one composer without turning labels into a path leak.

V1 file browsing does not allow Frontend path typing or pasted local paths. Users navigate and select through App Server-provided roots, directory entries, and opaque ids. This keeps raw path-shaped input out of Frontend state and avoids duplicating native picker behavior inside the shared UI. A future shell-native "open path" helper may accept raw input inside an App Shell and pass it directly to App Server for validation, but the raw path must not become Frontend state, durable product state, Chat metadata, diagnostics, or support export data.

File browser navigation treats allowed roots as hard boundaries. Breadcrumbs show only a safe root label plus relative segments below that root; they must not expose absolute path components or shell-specific parent locations. Parent or `up` navigation is disabled at the allowed root, and App Server never returns parent entries outside the allowed root. Changing filesystem scope requires an explicit new, continue, fork, or future add-root flow rather than breadcrumb traversal.

File browser and attachment validation canonicalize resolved filesystem targets before allowed-root checks. Symlinks whose resolved targets remain inside the Task Project Context or active allowed additional roots are allowed. Symlinks that resolve outside those roots are shown as unavailable in browse results or rejected with a structured validation error before send. OpenAIDE must not rely on textual path prefix checks alone for file browser entries, native picker handoff, attachment handle creation, or `task/send` validation.

For symlinked `fileReference` attachments, App Server uses the resolved canonical target for validation and duplicate detection. App Server preserves the user-selected path for Agent-facing reference delivery and workspace-relative display labels when the selected path is inside the Task Project Context or active allowed additional roots and the canonical target also passes allowed-root validation. If either the selected path scope or the canonical target scope fails, selection or `task/send` fails with structured validation instead of silently rewriting to the other path. This preserves user intent for valid workspace symlinks while preventing symlink escape.

If the same canonical file is selected through a different overlapping allowed root, App Server still returns `duplicateAttachment` and creates no second handle. Differing selected-root attribution may change safe label context for a newly selected non-duplicate file, but it is not a way to include the same canonical file twice in one composer.

V1 duplicate detection is path-based after normal resolution, not filesystem-identity based. OpenAIDE does not collapse hardlinks by inode/device identity. If the user selects two different hardlink paths that both pass validation, they remain distinct selected paths with distinct attachment handles. This keeps behavior portable across supported operating systems and avoids surprising path rewrites. A future filesystem-identity feature can add inode/device-aware duplicate detection if the product needs it.

If the selected symlink path is inside allowed roots but the canonical target is outside, OpenAIDE treats it as a symlink-escape validation error in v1. Frontend must not offer to add the symlink's parent as a fix because that still would not authorize the target. A future new/continue/fork flow may offer to add the real target parent only after App Server can present a privacy warning without exposing raw paths to browser Frontend, but v1 can simply block with a structured error.

Durable Chat, Task snapshots, and normal diagnostics must not store symlink canonical target paths. Symlink state, selected raw paths, and canonical target facts stay in App Server resolver or validation internals only for as long as needed. Durable sent attachment metadata stores safe selected-path display metadata, such as a safe label or project-relative label when available, plus normal attachment fields. Support Export may later include redacted or tokenized symlink diagnostics if useful, but never raw selected or canonical target paths by default.

The App Server-backed file browser shows hidden files and ignored files inside allowed roots in v1. It does not implement `.gitignore` filtering because agent work often needs configuration, dotfiles, generated metadata, or files excluded from source control. Hidden and ignored files are attachable by default when they pass the same allowed-root, file-vs-directory, capability, and blocked-internal validation as other files. App Server hard-blocks known product-internal secret paths regardless of allowed root membership, including OpenAIDE state storage, runtime endpoint records, runtime auth token material, and other OpenAIDE-owned internal storage. Product-internal denylist checks apply after canonicalization to both selected path facts and resolved target facts, so symlinks or aliases cannot expose OpenAIDE state, runtime, or auth internals. This built-in product-internal denylist is not user-configurable in v1; exposing OpenAIDE internals requires a future explicit support/debug flow with separate warnings and export rules, not a validation preference. It does not hard-block generic secret-looking files such as environment files, SSH keys, cloud credential files, or local configuration merely because of their name. Those files may be valid Agent context. OpenAIDE v1 has no generic sensitive-file warning based on filenames or heuristics; explicit `embeddedSnapshot` confirmation and allowed-root scope are the v1 safeguards. A later sensitive-file warning feature needs explicit product and UI policy, and must not become a hidden validation denylist. Later UI filters may hide or de-emphasize file classes for convenience, but they must not become the validation source of truth.

V1 attachments support regular files only. Directory entries are browse-only containers and cannot be attached as message parts. Users attach individual files, or they explicitly add a directory as an allowed root when creating, continuing, or forking into a new Task. Symlinks are attachable only when the selected path and resolved canonical target pass allowed-root and product-internal validation, and the resolved target is a regular file. FIFOs, sockets, devices, pipes, and other special filesystem entries are unavailable in browse results or blocked with structured validation errors before send. This avoids hangs, side effects, and undefined Agent adapter behavior. Directory-as-attachment is deferred until OpenAIDE designs recursion semantics, aggregate size caps, ignore/filter behavior, symlink handling, privacy warnings, and Agent adapter delivery.

Pre-send attachment handles and non-sendable attachment candidates are ephemeral resolver resources, not durable product state. Frontend explicitly releases them when the user removes a row, clears the composer, or unmounts the Task Page. Release is idempotent and scoped to the creating `clientInstanceId` plus `taskId`, so repeated cleanup, reconnect cleanup, or late cleanup after TTL does not create user-visible protocol noise. App Server also expires abandoned handles and candidates after a default 30-minute TTL for disconnects, crashes, reloads, and webview destruction. Until included in `task/send`, a handle or candidate must not appear in durable Chat or Task snapshots as committed work. Once sent, OpenAIDE persists only the safe message-part metadata allowed by the attachment kind; raw resolver details remain server-side and `embeddedSnapshot` file contents are not persisted by default.

The abandoned-resource TTL is not a maximum lifetime for handles or candidates that still belong to a live composer. While the owning initialized client is connected or inside reconnect grace and the Frontend still has the handle or candidate in live composer state, App Server must keep that resolver resource alive, even if another client's turn keeps the Task running longer than the default TTL. If the owning client disconnects past reconnect grace, reloads or remounts without live composer state, clears or unmounts the composer, or explicitly removes the row, normal release and abandoned-resource cleanup apply.

Keeping a resolver resource alive is cleanup bookkeeping, not attachment validation. The liveness, heartbeat, and TTL paths may update ownership or last-seen bookkeeping, but they must not stat, canonicalize, or read selected files, refresh per-row attachment capability decisions, capture `embeddedSnapshot` bytes, or emit composer row validation changes. Validation state changes only through defined validation passes such as selection, mode change, embedded confirmation, live open/reveal, manual refresh, capability-change refresh, post-turn idle refresh, or `task/send`. If the filesystem, Agent capability, or Task context changes while a draft waits, the row keeps its last known validation state until one of those passes runs.

Attachment release is batch-oriented. `attachment/release` accepts a list of pre-send handle ids or non-sendable candidate ids and returns per-id outcomes so Frontend can clean up row removal, composer clear, unmount, or reconnect state in one request. Unknown, already released, consumed, or expired ids are idempotent success or no-op outcomes. Ids that App Server can identify as owned by another `clientInstanceId` or `taskId` return scoped authorization errors and do not release that other client's resource. One id's outcome must not prevent release attempts for other ids in the same batch.

Pre-send attachment handles and candidates are scoped to the creating `clientInstanceId` and `taskId`. Before send, only the initialized App Shell client that created the resource may use, validate, release, confirm a candidate into a handle, request live open or reveal for a handle, or submit it. Another tab, window, webview, or App Shell client cannot send, open, reveal, or confirm a resource it did not create, even if it is subscribed to the same Task. After `task/send` succeeds, safe sent attachment metadata becomes part of the durable user message and appears in Task snapshots for all clients according to the normal Chat model.

Pre-send attachment handles and candidates may survive only a same-client transport reconnect where the Frontend still has the resource id in live composer state. They are not reload-durable. Browser reload, VS Code webview recreation, Desktop remount, Mobile remount, or Task Page unmount loses the Frontend composer state and must not reconstruct attachment handles or candidates from durable Chat, Task snapshots, or App Server state. The only exception is a locked same-client submitting recovery record for an already-issued `task/send`, which may retain the original send request handle ids solely to retry the same idempotency key and must not expose them as editable, removable, confirmable, openable, or revealable live attachment rows. App Server releases explicitly cleared resources and lets abandoned resources expire through the default TTL.

Live composer attachment chips may offer open-file or reveal-file actions only while the row is backed by a live pre-send handle owned by the same `clientInstanceId`. Frontend invokes App Server with the handle id, not a path. App Server revalidates `clientInstanceId`, `taskId`, handle liveness, allowed-root scope, and the owning client's current shell capability before routing an open or reveal request back to that owning client. If a different client asks to open or reveal the handle, the handle is stale, released, expired, or the owning client cannot open or reveal it, App Server returns structured state that Frontend renders without exposing raw paths.

Live open/reveal applies to sendable `fileReference` handles and confirmed `embeddedSnapshot` handles before send. Both handle kinds still reference a selected local file in App Server resolver state until send or cleanup. Open/reveal is therefore a convenience action on that selected file, not on the eventual Agent delivery representation. It must not capture embedded bytes, switch attachment mode, mark embedding confirmed, or make an unconfirmed candidate sendable.

Live composer open/reveal failure is non-blocking. If no capable shell is available, the target shell disconnects, or the shell action fails, App Server returns a structured action error and may cause Frontend to hide or disable the action until capability state changes. The attachment remains sendable unless normal attachment validation marks the handle stale or invalid. Open/reveal is a convenience affordance, not part of attachment send eligibility.

If live open/reveal validation detects that the handle is stale or invalid, App Server returns the action failure plus the updated handle validation state in the same response. Frontend updates the composer row from that state instead of discarding it because the user clicked a convenience action. The action is still non-blocking, but stale or invalid validation state becomes part of the normal composer state and will block `task/send` until the user removes, fixes, or reselects the attachment.

Non-sendable attachment candidates and invalid attachment rows have no v1 open-file or reveal-file action. Candidates are not sendable handles, and invalid rows are Frontend presentation state or failed validation outcomes. Their v1 affordances are recovery actions such as confirm embedded, remove, fix, start a scope-change flow, or reselect. If a client asks App Server to open or reveal a candidate id, invalid row, stale row, or unsupported id, App Server returns a structured validation error instead of routing a shell capability request.

If `task/send` races with an already-sent or late release for the same handle despite the submitting-state UI lock, the first durable App Server action wins. If release is processed first, App Server marks the handle released and a later `task/send` fails with a structured released or invalid handle error without committing a user message or turn. If `task/send` reserves or consumes the handle first, a later release is an idempotent no-op for that handle and the send continues. This prevents late row cleanup from deleting resolver state that an accepted send is already using, while still making explicit release effective before send begins.

`task/send` accepts attachment handles by id plus message-level attachment order only. Frontend must not send inline ranges, message-part placement, echoed attachment metadata such as size, display label, content hash, raw path, resolver facts, or inferred MIME type as source of truth. App Server looks up each handle, verifies `clientInstanceId` and `taskId` ownership, revalidates allowed roots and current capabilities, computes or verifies fingerprints for snapshot content, normalizes absent or whitespace-only text according to attachment-only capability, and constructs the durable safe message metadata committed with the user message as non-empty text first when present followed by attachments in user-selected order.

If attachment-only send is unavailable and a client submits absent or whitespace-only text with otherwise valid attachments, App Server returns a structured message-shape validation error and commits no user message or turn. This is not an attachment validation failure: the submitted handles remain usable for that same client according to the normal failed-validation rules until explicit cleanup, composer unmount, or abandoned-resource cleanup applies.

Pre-send attachment handles are single-use. On successful `task/send`, App Server consumes the handles into durable safe message metadata in the same committed send outcome. If the client loses the `task/send` response and retries with the same idempotency key and request fingerprint, App Server returns the existing `turnId` and current Task snapshot even though the original handles are already consumed. A different send attempt using a consumed handle fails with a structured attachment-handle error such as `attachmentHandleConsumed` or `attachmentHandleInvalid`; it must not create another turn.

Failed `task/send` validation does not consume attachment handles. If one message part fails validation, App Server returns structured validation errors, commits no user message or turn, and leaves all submitted handles usable for that same client until the user removes them, releases them, unmounts the composer, or abandoned-resource cleanup applies after live composer ownership is gone. Handles are consumed only after App Server commits the durable user message and initial turn state. This lets users fix invalid attachments without reselecting every file while the composer remains live.

Durable sent attachment records snapshot safe display metadata at user-message commit. The snapshot includes the render label, selected root context when needed, selected relative label when safe, attachment kind, sent mode, and size or media type when known. Chat rebuild renders this committed snapshot instead of recomputing labels from the current formatter, current filesystem state, current project state, or expired resolver state. This preserves auditability when later label formatting, project names, directory names, or root disambiguators change.

Sent attachment metadata must not retain raw resolver paths for future retry or reuse. `Reuse last prompt` is available only when the last user message has non-empty text; it may restore that text into the composer and show prior attachment metadata as non-reusable context, but it must not recreate attachment handles, raw paths, file URIs, or `embeddedSnapshot` bytes from Chat history. If the last user message was attachment-only, `Reuse last prompt` is not offered. App Server supplies prior attachment context in the Task snapshot or recovery payload; Frontend renders it as provided and must not reconstruct recovery semantics by parsing Chat. That context contains render-ready safe metadata plus source `messageId` and `sentAttachmentId` identifiers so UI can display it directly while still tracing it to durable Chat. Recovery UI may still show prior safe attachment metadata, including the same safe delivery status visible in Chat when known, as non-reusable context. If the last user message had text plus attachments, reuse restores only the text. If the user wants to send the same files again in v1, they reselect them through normal attachment selection and App Server creates new pre-send handles under the current Task validation rules. `Select same files again` is not a v1 feature. Any future same-files action must be a separate explicit attachment selection flow, not a side effect of prompt reuse, and must design path retention, privacy warning, stale validation, shell routing, and failure states. This prevents normalized Chat history from becoming a hidden local path archive.

Recovery attachment context preserves the committed Chat source order. If the payload contains attachments from multiple source messages, App Server groups them by source `messageId` in durable Chat order. Within each source message group, attachments remain in committed user-message part order. Frontend must not reorder recovery attachments by delivery status, kind, label, size, or shell display heuristics, because recovery context is meant to help users recognize exactly what was sent.

When a committed user message has no non-empty text and contains attachments, Chat renders an attachment-only user message. It does not render an empty text bubble, invisible spacer text, or synthesized placeholder prompt such as "Sent attachments." The ordered sent attachment chips or cards are the message body. Normalized Chat and render-state rebuild must preserve that this was attachment-only rather than inventing fake user text, because synthetic prompt text would become misleading durable Chat content and could leak into reuse, export, copy, or Agent-history flows.

Copy, export, support export, and any future share operation for an attachment-only user message preserve the part structure. They must not synthesize prompt text, convert the message to a fake sentence, or include placeholder text to make the message look text-like. Target formats may include safe sent attachment metadata, such as the same safe labels and statuses visible in Chat, subject to the target format's redaction rules. They must not include raw paths, resolver ids, file URIs, raw root metadata, embedded file contents, `embeddedSnapshot` bodies, or full content hashes unless a future explicit export mode is designed with a user warning and opt-in.

Chat renders sent attachments as safe non-reusable chips or compact cards inside the user message. The renderable metadata may include safe display label, attachment kind, size or media type when known, sent mode such as `fileReference` or `embeddedSnapshot`, and availability or delivery status when known. It must not include absolute paths, raw file URIs, raw resolver ids, raw content hashes, raw content, embedded file body previews, or a resend-from-history affordance. Sent attachment chips have no v1 open-file, reveal-file, reopen-from-history, or `Select same files again` action because durable Chat does not retain the raw resolver data required to do that safely. `embeddedSnapshot` chips may show a short note that content was sent but not retained. A future same-files action may start a new App Server attachment selection flow and create new pre-send handles only after explicit design for path retention, privacy warning, stale-file validation, shell routing, and failure states; it must not reuse historical metadata as file access.

OpenAIDE stores both turn-level state and safe per-attachment delivery status. Turn-level state records whether the turn is running, completed, failed, interrupted, or canceled. Each sent attachment chip may separately report `sent`, `deliveryFailed`, or `deliveryUnknown` when App Server can distinguish those outcomes. The same safe status values may appear in recovery context for prior attachments, but only as explanatory non-reusable metadata. For ACP, `sent` means the attachment part was included in the `session/prompt` payload accepted for write by OpenAIDE's Agent transport, not that the Agent, model, or tools semantically used it. If transport fails before or during write, App Server records `deliveryFailed` when failure is clear or `deliveryUnknown` when it cannot know whether the Agent received the payload. ACP `session/prompt` success and turn completion do not upgrade attachment status to `used`, `read`, or `processed`; only a future explicit Agent protocol signal can report per-resource consumption. This prevents UI from claiming every file reached the Agent when only text was delivered, one attachment failed, the adapter failed before delivery confirmation, or the model ignored provided context.

Sent attachment metadata and delivery status changes are normalized Chat events, not render-only cache. `openaide_core::chat_history` persists safe attachment metadata and status events so reload and render-state rebuild produce the same visible chips/cards and statuses. Those events must not include raw resolver paths, file URIs, raw content hashes, `embeddedSnapshot` bytes, shell picker details, or raw ACP envelopes.

The initial user-message commit includes safe sent attachment metadata with a default status such as `pendingDelivery` or `deliveryUnknown`. Attachment delivery status changes are later normalized Chat events persisted before state-stream emission. If the process exits after the user-message commit but before a delivery status event is persisted, reload shows the user message and safe attachment chips with `deliveryUnknown` plus the appropriate interrupted or failed turn recovery state. App Server must not invent a later delivery result during recovery.

The committed user-message event assigns durable App Server-owned `sentAttachmentId` values to each sent attachment. Pre-send attachment handle ids remain transient resolver handles: they are scoped, TTL-bound, single-use, and never used as durable chat identity. Attachment status events and render-state rebuild reference `sentAttachmentId`, which lets handles be consumed, released, or garbage-collected independently of persisted chat history.

Attachment delivery status transitions are monotonic and idempotent. `pendingDelivery` or `deliveryUnknown` may move to `sent` or `deliveryFailed` only from durable adapter evidence. `sent` is terminal for delivery and must not be downgraded by later Agent, model, or tool failures, because those are usage failures rather than payload delivery failures. Duplicate or replayed delivery status events are ignored by stable event identity and ordering so normalized Chat replay is deterministic.

If a delivered `fileReference` reaches ACP as a `resource_link` and the Agent later fails to read that file, OpenAIDE treats that as Agent/tool activity or turn failure, not attachment delivery failure. The attachment chip remains `sent` because prompt payload delivery succeeded. App Server must not retroactively mark delivery failed unless the `session/prompt` payload or adapter delivery itself failed before the Agent received the reference.

After a successful send, App Server keeps raw resolver details for sent `fileReference` only as long as needed for active Agent delivery and immediate in-process bookkeeping. Durable storage, normalized Chat, Task snapshots, diagnostics, and Support Export keep safe sent metadata only. If a future feature needs to reopen, reveal, or redeliver sent files from history, it must be designed as an explicit App Server capability with its own privacy, path-retention, stale-file validation, and shell-routing rules instead of piggybacking on Chat history.

If Agent delivery fails after `task/send` commits but before a `fileReference` reaches ACP delivery, App Server must not automatically reuse old raw resolver details for retry. The turn becomes failed or interrupted with the durable user message and safe sent attachment metadata. User-initiated retry or `Reuse last prompt`, when available, can reuse non-empty text only, but files must be reselected to create new pre-send handles under current validation rules. This matches the no-implicit-retry rule and avoids stale or hidden file access after resolver cleanup.

Attachment delivery failure remains visible in Chat. App Server keeps the durable user message and safe attachment chips/cards visible, marks the turn failed or interrupted with a local event such as attachment delivery failed, and includes stable error kind/data for UI actions and support diagnostics. Chat must not remove the user message, hide the attachment chips, or imply that the Agent received `fileReference` or `embeddedSnapshot` content that did not reach the Agent adapter.

For ACP Agent adapters, OpenAIDE message parts are translated to ACP `ContentBlock[]` only at the adapter boundary. ACP requires prompt text support and baseline `resource_link` support, while image, audio, and embedded resource contents are gated by Agent `promptCapabilities`. `fileReference` maps to ACP `resource_link` when the Agent can access that local path through the Task Project Context or active additional roots supplied to the Native Session. Because ACP `resource_link` needs a usable URI for the Agent, App Server may generate and send a real local file URI or raw local path only to the local Agent process at this adapter boundary after validation. That URI or raw path must not appear in App Server Protocol snapshots, Frontend state, Task snapshots, normal logs, diagnostics, or Support Export. `embeddedSnapshot` maps to ACP embedded `resource` only when the initialized Agent advertises `promptCapabilities.embeddedContext`; its adapter-delivered metadata is limited to safe metadata such as `sentAttachmentId`, safe display label, text or media type, size, and stable-per-sent-attachment synthetic non-file resource identity. If an adapter can deliver embedded text but lacks resource metadata fields, App Server must not mutate the text with synthetic metadata headers. Otherwise `task/send` fails validation or the user must choose `fileReference` when that is valid. OpenAIDE therefore keeps internal attachment references protocol-safe and capability-marked, then maps them to ACP text, resource links, images, audio, or embedded resources only after validating the initialized Agent capabilities.

`task/send` returns after the user message and new turn are durably accepted, not after the Agent turn finishes. The handler validates that the Task exists and is not already running, acquires or verifies live Native Session ownership, generates a new `turnId`, commits the user message and initial turn state through core, enqueues or starts app-server `turn_runner`, and returns the `turnId` plus an updated `TaskSnapshot` only after state is coherent enough that reload shows the submitted message. `task/send` requires a client-generated opaque idempotency key scoped to the Task and send attempt. App Server persists the key-to-`turnId` mapping and a request fingerprint in the same durable transaction as the user message and new turn. If a client loses the response and retries with the same key and same fingerprint, App Server returns the existing `turnId` and current Task snapshot without creating another turn. Reusing the same key with a different prompt, options, attachments, or other send-affecting request content returns structured `idempotencyKeyConflict`; App Server must not silently return the old turn or create a new one. A different key creates a distinct send, even when the prompt text is identical, because repeating the same text can be intentional. Idempotency keys are retained as turn metadata for as long as the Task exists; deleting the Task deletes its keys, and OpenAIDE does not use a separate timed cleanup policy for send keys. Agent output, completion, cancellation, and failure stream afterward through `state_stream` as Task events. Long-running turns must not tie up the request/response lifecycle. If Agent delivery fails after the user message is persisted, the turn becomes failed or interrupted with a visible local event; App Server must not erase the submitted message or implicitly retry it.

For `embeddedSnapshot`, the persisted canonical `task/send` fingerprint includes the full raw-byte content hash captured at send time. That full hash is internal App Server storage/runtime state used for idempotency precision, conflict detection, and local send-material bookkeeping. It is not a client-provided field and must not be exposed to Frontend, Chat, normal logs, or Support Export except through the redacted or truncated correlation forms described in the diagnostics rules.

For `fileReference`, the persisted canonical `task/send` fingerprint excludes file contents and content hashes. The fingerprint treats the attachment as a validated live reference to be resolved by the Agent through the Task Project Context or allowed additional roots, not as a snapshot of bytes.

OpenAIDE App Server generates `turnId` before sending a user prompt to the Agent. `turnId` identifies one user-initiated Agent run inside a Task and is distinct from ACP ids. `turn_runner` stamps every accepted normalized event with `(taskId, turnId)` and uses that identity for stale-response guards. If an ACP Agent exposes its own turn, request, or update ids, OpenAIDE stores them only as correlation metadata rather than product identity.

`task/cancel` is best-effort cancellation with explicit local final state. It requires `taskId` and may include `turnId`; when `turnId` is omitted, App Server targets the current active turn if one exists. The method is idempotent if that turn is already canceled or no longer running. `turn_runner` stops accepting future updates for the targeted `turnId`, asks `agent_runtime` to send ACP cancel or interrupt when supported, and, if ACP cancel is unsupported, closes or restarts the Native Session only when needed. Core records the turn as canceled or interrupted locally, appends a normalized local cancellation event to Chat, and App Server emits Task state with no active turn. Late ACP updates for the canceled `turnId` are dropped or stored only as diagnostics; they are never rendered into Chat.

OpenAIDE does not implicitly retry Agent turns in v1. If a turn fails because of transport crash, Agent failure, storage failure, or App Server restart, core marks the turn failed or interrupted with a visible local event when it can persist one and keeps the user message in Chat. App Server must not automatically resend the prompt because doing so could duplicate file edits, terminal commands, tool calls, or permission effects. OpenAIDE must not automatically roll back, undo, or compensate external Agent side effects such as file edits, terminal commands, tool calls, permission effects, or MCP/tool actions. Those effects are outside App Server's durable state and are not safely reversible. A user-initiated retry creates a new `turnId`, may reuse the same prompt content and Task or Native Session when valid, and continues to ignore stale updates from the previous `turnId`. Remediation after failure is user-driven; users inspect project diff, terminal state, logs, or support export diagnostics and decide the next action.

Agent selection is per Task at creation time, with an app-wide default Agent remembered by App Server. OpenAIDE should not use hidden per-Project Agent defaults in v1.

Configuration Options belong to the current Agent Native Session. OpenAIDE should render the current option state reported for that session, but v1 should not cache or restore previous Configuration Option values across Tasks.

The Agent-owned session list is the source of truth for Task history. App Server uses ACP `session/list` and `session_info_update` where available to discover sessions, titles, working directories, and recency, then projects those sessions into OpenAIDE Tasks. OpenAIDE may persist a local task/session list projection and OpenAIDE-owned overlay metadata, such as archive state or shell presentation state keyed to the Agent/session identity.

OpenAIDE keys projected Tasks, overlays, local normalized Chat history, and active routing by `(agentId, sessionId)`. ACP `sessionId` values are treated as unique only within the owning Agent.

Normalized Chat history is separate from session-list metadata. OpenAIDE should persist normalized Chat history for Tasks so reloaded Chat can look the same, support resume-capable Agents, and recover quickly when local history is valid, but this history does not decide whether a Task exists in the sidebar. Session list controls Task existence and list metadata; normalized Chat history controls Chat rendering for an opened Task. OpenAIDE should not persist full raw Agent session data or Configuration Option values. Earlier persistence decisions that stored raw or overly broad Agent data are replaced by this normalized Chat architecture.

App Server refreshes Agent-owned session lists on app start and when relevant filters or Agent connections change, listens for ACP `session_info_update` while connected, and exposes a manual refresh action. OpenAIDE should avoid aggressive polling except for explicit recovery cases such as reconnecting after sleep or a lost Agent connection.

For resume-capable Agents, persisted normalized Chat history must restore the same user-visible Chat after reload. It should include the rendered conversation structure needed for continuity, such as part-based user messages, agent messages, folded tool activity, permission outcomes, terminal or file activity summaries/details that were visible in Chat, errors, and final turn state. It must not persist raw ACP envelopes, secrets, transport logs, private Agent metadata, or Configuration Option values.

Opening an existing Task from history uses ACP `session/load` when Chat history should be replayed by the Agent. For Agents and sessions where OpenAIDE has persisted normalized Chat history and the Agent supports ACP `session/resume`, opening the Task can restore local Chat from OpenAIDE storage and call `session/resume` to reconnect without replay. Calling `session/load`, `session/resume`, sending a prompt, cancelling, or consuming a live update stream requires live Native Session ownership, because the Agent owns the real session state and may diverge from OpenAIDE's local projection. If an Agent supports both paths, App Server prefers `resume` when valid local normalized Chat history exists and ownership is acquired, and falls back to `loadSession` when local history is missing, corrupt, version-incompatible, or cannot be trusted after another owner epoch. App Server chooses `load` or `resume` from Agent capabilities, ownership state, and local history validity; Frontend only renders the resulting Task state.

If a session appears in `session/list` but OpenAIDE has no valid normalized Chat history for it, App Server opens it with `session/load` when the Agent supports loading. If the Agent only supports `session/resume`, opening that Task shows a recoverable Chat-history-unavailable error instead of inventing an empty Chat.

If local normalized Chat history exists for a session that no longer appears in the Agent-owned `session/list`, OpenAIDE deletes that local Chat history. Missing sessions do not remain in the sidebar and are not kept as orphaned recovery entries in v1.

Archive is a local OpenAIDE overlay keyed to Agent/session identity. Archiving hides a Task from the default OpenAIDE list without calling ACP `session/delete` or changing the Agent-owned session list. Destructive delete is a separate feature shown only for Agents that advertise ACP `sessionCapabilities.delete`.

Multiple App Server processes may open the same OpenAIDE state root so different App Shells can see the same persisted Agents, Projects, Tasks, and normalized Chat history without requiring a durable daemon. Shared storage is protected by the storage layer: SQLite uses normal transactional concurrency, schema migrations and startup repairs use short storage-level locks, and live Agent interaction uses a live Native Session ownership lock keyed by Agent Identity and Native Session id. OpenAIDE does not use a global lifetime state-root writer lock in v1.

Each App Shell is responsible for deciding when it needs a local App Server for the selected OpenAIDE state root, but reusable launcher/client code owns the attach-or-launch mechanics. The shared launcher interface resolves the state root, discovers an existing compatible AppServer endpoint, validates the endpoint, handles local auth token material, connects over the best requested local transport, initializes through App Server Protocol, and reuses that process. If no compatible process is reachable, the launcher may acquire a short launch lock, start AppServer, publish its local endpoint, and then initialize as a client. If several App Shells race to launch, one process should win and the others should attach to it; starting a separate AppServer remains a fallback only when attach/launch discovery cannot prove a reusable process exists. Multiple App Shell clients, such as Desktop, Web App, Mobile App, and VS Code Extension, may attach concurrently to the same AppServer process. App Shells and Frontend must not open the core SQLite database directly or act as independent storage writers outside App Server. App Server and clients are expected to use the most efficient available local transport for that shell while preserving the same App Server Protocol product API. Native App Shells can use native IPC, pipes, or process channels.

AppServer discovery uses a per-state-root runtime endpoint record, not a fixed global port or shell-specific registry. Endpoint records live under the operating system runtime/cache location keyed by a state-root fingerprint, not inside the durable OpenAIDE state root or core SQLite database. Product state stays in the state root; process coordination stays in runtime storage so endpoint records do not pollute backups, sync, support exports, or copied state roots. The record contains enough local-only data for the launcher to probe a candidate process, such as server id, pid, start time, available transports, endpoints, canonical state-root facts, state-root fingerprint, an optional lifecycle hint, and one high-entropy local auth token or token reference minted for that AppServer process. The process-scoped token authenticates transport access only; it is not user identity, App Shell identity, or product state. `clientInstanceId` remains the initialized client identity after transport access succeeds. The token is valid only for the AppServer process lifetime, is stored only in the protected runtime endpoint record or equivalent protected runtime material, and is removed or invalid after process exit. Stale tokens in stale records fail probe or `initialize` and cause record cleanup. AppServer may expose multiple attach transports at once so a process launched by one shell can be reused by another shell. Stdio can be used as a parent/launch transport, but it must not be the only reusable endpoint when cross-shell reuse is expected. AppServer should always expose at least one attachable local transport for native shells, and may expose a browser-safe loopback transport such as websocket or HTTP plus SSE when the Web App needs to attach. The launcher chooses the best compatible transport from the endpoint record. Browser-exposed transports bind to loopback by default and require local auth plus Origin checks. The record is protected with local filesystem permissions and is not product state. Endpoint-record lifecycle is a best-effort hint only; live probe and `initialize` responses are authoritative. The launcher must verify that the process is alive, the endpoint responds, the server proves it owns the expected state root, and the protocol is compatible before reuse. A `serverStopping` initialize error means the launcher ignores that endpoint and attaches to or starts another process. Stale, unreachable, or invalid records are cleaned up. Endpoint records, auth tokens, ports, socket paths, and machine-specific details must be redacted from logs, diagnostics, support export, source, tests, docs, UI text, comments, package metadata, and commit messages unless represented only as generic examples. Auth tokens must not be written to product state or persistent Frontend storage.

AppServer is an on-demand local process for a selected state root, not a durable background daemon in v1. Its lifetime is tied to initialized App Shell clients rather than raw transport connections, explicit leases, or any individual Frontend view. A transport connection does not count for lifetime until the client successfully calls `initialize` and `client_hub` records an initialized client identity. Browser tab reload, VS Code webview recreation, Desktop window recreation, Mobile view remount, or one Task Page closing should not stop an AppServer while at least one initialized App Shell client remains connected or is inside reconnect grace. In the Web App, the Web App shell process owns that lifetime client and keeps it alive while the Web App process runs; browser tabs are volatile Frontend views that may also initialize product protocol clients for state subscriptions, task-scoped request routing, and view-scoped pending state, but tab close or reload is not the Web App lifetime signal. Default reconnect grace is 10 seconds. Disconnect detection is driven by transport close, heartbeat timeout, or request/session timeout depending on transport; heartbeat or transport timeout should be longer than reconnect grace, usually around 15-30 seconds for browser transports. Explicit `client/disconnect` may exist for native shell close and best-effort browser unload, but AppServer must not depend on it for correct lifetime because browser unload events, mobile suspension, and forced process exit are unreliable. When a transport drops or heartbeat expires, `client_hub` starts reconnect grace for that initialized client identity; if the same client reconnects and reinitializes before expiry, lifetime continues. When no initialized App Shell clients remain after grace expiry, AppServer enters graceful shutdown. Reconnect grace and heartbeat timeout can be overridden for development or constrained runtime environments through CLI, environment, or runtime config, but they are not product Settings in v1. If an App Server exits while one of its Tasks is running, live streaming for those Tasks stops; a later App Server reconciles from Agent session list and local normalized Chat where possible, subject to the Agent's `load` or `resume` capabilities.

AppServer process lifecycle uses three states: `running`, `draining`, and `stopping`. `running` accepts initialized clients and normal App Server Protocol work. `draining` starts after all initialized clients are gone and reconnect grace has expired; it rejects new task-starting work but still accepts `initialize` from a compatible client for the same state root. Compatible `initialize` during `draining` atomically aborts shutdown, registers the initialized client, returns AppServer to `running`, and returns a normal initialize response with `serverLifecycle: "running"`. AppServer does not return a successful half-attached `draining` state to clients. Incompatible initialize during `draining` returns a structured protocol error. `stopping` starts after irreversible teardown begins, such as closing Agent transports, releasing live Native Session ownership, or removing the endpoint record. `initialize` during `stopping` returns a structured `serverStopping` error. A `stopping` AppServer is not reusable; launchers must ignore it and attach to or start another process. Reconnect grace should avoid most `draining` races, but the explicit state still prevents a reload or late shell attach from creating a second AppServer while the old process still owns live Agent work.

Graceful App Server shutdown attempts to detach or close cleanly rather than implicitly cancel its Running Tasks. Shutdown ordering is release-last for live Native Session ownership: App Server stops accepting new client requests or new Agent work, stops accepting new ACP updates except a bounded drain of updates already accepted by `turn_runner`, persists accepted normalized Chat events and Task interrupted or detached state, moves pending live requests to interrupted or detached memory state, asks Agent connections to detach or close cleanly if ACP supports it, closes the Agent process or transport when no clean detach path exists, releases live Native Session ownership leases, removes or marks unavailable the endpoint record, and finishes stopping. Releasing live ownership before persisted Chat and Task state are coherent is forbidden because another AppServer could acquire ownership and recover against stale local state. Shutdown must not call destructive session delete, and explicit cancellation remains a separate user intent. App Server exposes renderable shutdown-risk state, such as whether closing the last initialized client would interrupt or detach Running Tasks, but App Shells own the close warning before disconnecting because browser tabs, Desktop windows, VS Code windows, and Mobile surfaces have different close affordances. App Server must not rely on sending a last-second confirmation request to a disappearing client. If the shell cannot warn, the user forces close, or the process is killed, AppServer still enters graceful or unclean shutdown rather than leaving a hidden durable server. On a later launch, App Server reconciles using Agent-owned `session/list` plus local normalized Chat, then uses `resume` or `load` where supported.

If graceful shutdown cannot persist accepted Chat events or coherent interrupted, detached, completed, failed, or idle Task state, App Server retries or flushes for a bounded shutdown window and records diagnostics suitable for redacted support export. It should remove or mark unavailable its endpoint record so new clients do not attach to a dying process, and it should close or detach Agent transport as best effort. It must not explicitly release the live Native Session ownership lease as a clean release. The lease is left to expire, and the next AppServer treats takeover as unclean detach or crash recovery. This preserves the meaning of clean lease release: local state was made coherent before ownership became available.

The same rule applies to live storage loss during a Running Task. If App Server can no longer persist accepted Chat events, Task state, or ownership heartbeats, it must stop live rendering and Agent update acceptance instead of continuing with memory-only UI. Connected clients receive storage error state where possible. Agent transport is closed or detached as best effort, and live ownership is recovered by lease expiry if coherent state cannot be persisted. App Server must not attempt automatic rollback or compensation for file edits, terminal commands, tool calls, permission decisions, or other Agent side effects that may already have happened before storage failed.

Crash or kill recovery is treated as unclean detach. On launch, App Server reads durable state, clears or expires stale live Task ownership records, marks abandoned Running Tasks as interrupted or detached, connects enabled Agents, refreshes Agent-owned `session/list`, and reconciles Tasks from that source of truth. Local normalized Chat history for sessions missing from the Agent-owned session list is deleted. For Tasks that still exist, App Server may hydrate durable Chat for display when a user opens the Task, using local normalized Chat, `session/load`, or `session/resume` according to Agent capabilities and history validity, but it must not automatically resume an abandoned live turn, resend the last prompt, or present the Agent as still working from before the crash. Continuing after interruption is an explicit user action that creates a new turn or calls an Agent-specific resume path only when the product can represent the result coherently. If no path can recover Chat for display, App Server exposes a recoverable unavailable state rather than an empty Chat. Recovery must never invent a completed or failed final Agent message. Storage must persist enough active-turn metadata to distinguish a cleanly completed turn from an App Server exit during an active turn, including whether the turn was interrupted while waiting for permission or user input.

When a user opens an interrupted or detached Task, Frontend renders the durable Chat up to the last persisted event plus a local interruption marker from the Task snapshot. App Server owns the available recovery actions and any prior attachment context in the snapshot or recovery payload. Prior attachment context contains render-ready safe metadata plus source `messageId` and `sentAttachmentId` identifiers, not raw paths, handles, resolver ids, or reuse-capable data. Prior attachment context preserves durable Chat order by source message and committed user-message part order within each message. Frontend renders that payload and must not derive reuse eligibility, attachment context, action availability, or attachment order by parsing Chat. `Continue` is the primary action and starts a new explicit continuation path; `Reuse last prompt` is shown only when the last user message contains known non-empty text, that text is still valid for the Task, and App Server marks it safe to reuse. UI may label this action `Retry last prompt` when that is clearer, but the product and protocol semantics are reuse and prefill, not automatic retry. No recovery action auto-sends on page open, restored tab mount, or subscription refresh. The user can inspect project state, terminal state, Chat history, and diagnostics before choosing an action.

`Continue` for an interrupted or detached Task acquires live Native Session ownership, opens or resumes the Native Session as needed for future interaction, clears the recovery affordance into normal active-composer state, and focuses the prompt box. It does not send text, retry a previous prompt, or mark the interrupted turn completed. The next Agent turn starts only when the user submits a new prompt through the normal send path.

`Reuse last prompt` for an interrupted or detached Task acquires live Native Session ownership if needed, opens or resumes the Native Session as needed for future interaction, loads the last known non-empty user prompt into the active composer, and focuses the prompt box. It does not create a turn, append Chat, send text, or mark the interrupted turn completed. If the last user message was attachment-only, this action is not offered; App Server-provided recovery context may show prior safe attachment metadata, including the same safe delivery status visible in Chat when known, as non-reusable context while users must reselect files through normal attachment selection in v1. If the last user message had text plus attachments, reuse restores only the text and shows prior attachment metadata as non-reusable context. `Select same files again` is not a v1 action; any future same-files action must be a separate explicit attachment selection flow, not prompt reuse. The reused prompt becomes durable only if the user submits it through the normal send path.

Composer draft text is Frontend-owned ephemeral presentation state. App Server may return a client-scoped one-shot prefill payload for actions such as `Reuse last prompt`, and the receiving Frontend applies it to local composer state. App Server does not persist unsent drafts, synchronize drafts between App Shell clients, include drafts in Task snapshots, or treat drafts as Chat. `task/send` is the boundary where text becomes durable product state and enters the normal turn lifecycle.

Startup recovery is lazy. App Server reconciles lists, connects enabled Agents, refreshes history, and marks interrupted or recoverable Tasks, but it does not automatically call `session/load` or `session/resume` for every interrupted Task. Recovery of an individual Task happens when a user opens it or a client subscribes to its Task snapshot. This prevents startup from loading many old Tasks or surprising the user. If an App Shell restores open tabs or windows, those restored views request Task snapshots and may trigger display hydration for visible Tasks, but they must not continue abandoned turns without explicit user intent.

Task recovery is split by meaningful workflow modules. `openaide_core::startup_recovery` detects previously Running Tasks after crash or shutdown and marks them interrupted or recoverable. `openaide_core::task_opening` decides how to open one Task, including local Chat validity, `resume` versus `load`, and recoverable unavailable state. `openaide_core::chat_history` validates and returns normalized Chat. `openaide_core::tasks` owns Task status and projection updates. App-server `agent_runtime` performs ACP `resume` or `load` I/O. App-server `history_sync` only triggers list refresh and passes session-list snapshots into `openaide_core::task_history_sync`.

For v1, the Web App UI is not served by App Server. The Web App has a real App Shell/bootstrap layer because browser JavaScript cannot read OS runtime endpoint records or launch App Server directly. The Web App Shell invokes `openaide-app-server-client` or equivalent shared attach-or-launch code for the selected state root, obtains ephemeral browser-safe connection info, and then serves or bootstraps the shared Frontend. The browser receives App Server endpoint and token material only through this bootstrap path, such as initial page data or a local bootstrap endpoint, and keeps it in memory or session storage rather than durable product state. After bootstrap, the browser connects directly to App Server over the selected browser-safe transport and speaks the normal App Server Protocol. Each Web App tab calls `initialize` with `shellKind: "web"` and a tab-scoped `clientInstanceId`. The Web Frontend generates a high-entropy `clientInstanceId` on first boot and stores it in browser `sessionStorage` so reload reuses the same tab identity while separate tabs remain separate initialized clients. If `sessionStorage` is unavailable, the Web Frontend keeps the id in memory and a reload becomes a new initialized client. The Web App must not store `clientInstanceId` in `localStorage`, product state, durable settings, or a shared bootstrap value that makes multiple tabs share one client identity. The Web App Shell/bootstrap process does not call `initialize` as a product client unless it later has its own visible control surface. If all Web App tabs close or disconnect and reconnect grace expires, AppServer can shut down even if the Web App Shell is still serving assets. The Web App Shell must not proxy normal App Server Protocol traffic or own product protocol routing unless a later explicit design changes that. App Server keeps product workflow behind the same App Server Protocol. OpenAIDE's Web App is a local-first shell for browser-based use, not a cloud service, team server, or multi-user product surface. Remote preview domains used during development must remain outside source, docs, package metadata, and product behavior.

The Web App development command starts the Web App Shell/bootstrap layer, Frontend server, and App Server attach-or-launch flow together. Release packaging may bundle or install built `packages/frontend` assets alongside the Web App Shell and App Server launcher, but App Server source must not directly depend on TypeScript source paths; build and packaging scripts own asset wiring.

Initial shell transports are: VS Code Extension starts or attaches to `openaide-app-server` through shared attach-or-launch and bridges webview messages through the extension process; Desktop starts or attaches through shared attach-or-launch over stdio, a local socket, or native IPC; Web App Shell starts or attaches through shared attach-or-launch and bootstraps the browser with connection info for a configurable local browser-safe App Server transport. Web App should support websocket and at least one non-websocket browser transport such as HTTP plus SSE. Transport choices live in `openaide-rs/app-server-transport` and shell adapters, not in shared Frontend rendering code.

Transport selection is runtime/developer configuration through CLI flags, config, or environment, not ordinary product Settings in v1. Most users should not need to choose transport mechanics; constrained environments can override them explicitly.

Network and browser transports should expose lightweight health endpoints such as `/readyz` and `/healthz` for startup, proxy, and diagnostics workflows. They should bind to loopback by default and enforce Origin/auth checks for browser-exposed endpoints. Stdio transports do not need HTTP health endpoints.

Remote preview through a user-managed proxy or tunnel is not an official product mode in v1. OpenAIDE supports a local-first Web App and configurable transports for development or constrained personal setups, but source, tests, docs, UI text, comments, package metadata, and commit messages must not encode private preview domains or machine-specific proxy setup.

Persisted OpenAIDE state may be opened by multiple App Server processes on the same machine. Shared sessions and Agents are a persisted-storage property, not a daemon requirement. Storage must tolerate concurrent local readers and short writes from separate App Server processes while preserving product invariants through explicit transactions, schema migration locks, startup repair locks, and live Native Session ownership lease rows.

App Server owns the persisted state layout and storage invariants. App Shells supply the root location appropriate to the shell and operating system, such as an extension storage location, desktop app data location, or local web shell state directory. Frontend must not decide filesystem paths or storage layout.

`openaide_core::storage` uses SQLite for core state: Projects, Agent definitions and settings metadata, Task projections and overlays, normalized Chat events, materialized Chat render state, active-turn metadata, live Native Session ownership, and migrations. SQLite is required because OpenAIDE needs atomic multi-table updates for Task creation, session-list reconciliation, Chat event append plus render-state update, and Agent identity cleanup. Plain JSON files should not hold core mutable product state. Files remain appropriate for large support exports, logs, diagnostics, and future binary attachments.

Live Native Session ownership uses SQLite lease rows as the v1 product-visible mechanism, keyed by `(agentId, sessionId)`. A lease row records owner AppServer id, process id and process-start identity when available, ownership epoch, acquired time, heartbeat time, and lease expiry. Default owner heartbeat interval is 5 seconds, and default lease expiry is 30 seconds after the last heartbeat. Ownership acquire, heartbeat/update, stale-owner expiry, and release happen in short storage transactions. OpenAIDE must not hold a SQLite transaction or lease acquisition transaction across Agent I/O, App Shell capability requests, or other unbounded waits. The owning AppServer heartbeats while it holds live ownership. A new AppServer may expire a stale owner only after lease timeout and, when possible, a process-death or process-start mismatch probe. Forced takeover without process-death proof is allowed only after expiry and must be treated as unclean detach during recovery. The ownership epoch is written into Chat validity metadata so stale local projections can be detected before future live interaction. OS file locks may be added later as an implementation hardening layer, but they are not the v1 source of truth because they do not provide enough recovery metadata, UI explanation, or migration-friendly state.

Clean release of a live Native Session ownership lease is allowed only after the owner has persisted all accepted Chat events and the corresponding Task interrupted, detached, completed, failed, or idle state. If the owner cannot make local state coherent before exit, it must not perform a clean release; recovery must treat the next owner as taking over after unclean detach rather than as a clean lease release.

SQLite access is blocking local I/O behind a dedicated storage executor or worker owned by `openaide_core::storage`. App-server never touches SQLite directly. Workflow modules open explicit transactions for atomic product changes. Each App Server process serializes its own storage writes through storage, while SQLite WAL mode, busy timeouts, and transaction boundaries arbitrate cross-process local access. OpenAIDE must not hold SQLite transactions across Agent I/O, App Shell capability requests, or other unbounded external waits.

The initial SQLite crate choice is `rusqlite` without an ORM. Storage code should use direct SQL with explicit transactions and schema control. `tokio::task::spawn_blocking` or a dedicated storage thread/channel can isolate blocking I/O from async app-server work. Migration tooling, such as `refinery` or `rusqlite_migration`, should be selected during implementation after comparing fit. `sqlx` and ORM layers are deferred unless direct SQL stops carrying its weight.

The package layout should express these boundaries. The initial target structure uses a Codex app-server-like Rust crate family: `openaide-rs/` for Rust crates, `openaide-rs/app-server` for the shared rich-client server, `openaide-rs/app-server-protocol` for protocol types and schema generation, `openaide-rs/app-server-transport` for stdio, local socket, websocket, or browser transports, `openaide-rs/app-server-client` for reusable attach-or-launch, endpoint discovery, endpoint probing, transport selection, and protocol client helpers, `openaide-rs/core` for product state and workflow, and `openaide-rs/acp` for Agent Client Protocol integration. Rust package names should be globally clear, such as `openaide-app-server`, `openaide-app-server-protocol`, `openaide-app-server-transport`, `openaide-app-server-client`, `openaide-core`, and `openaide-acp`. Use the `app-server-*` prefix only for crates that are specifically part of the client-facing App Server boundary; reusable product logic and Agent integration crates should not be made app-server-specific by name. `openaide-rs/app-server-daemon` is deferred until there is a real daemon need. `packages/frontend` remains the shared render-only Frontend. `app-shells/vscode`, `app-shells/desktop`, and `app-shells/web` are App Shell packages; `app-shells/web` owns the Web App bootstrap layer for browser-safe attach-or-launch and Frontend bootstrapping. Existing names that imply a VS Code-first or webview-first architecture should be replaced during the refactor instead of preserved for compatibility.

The first structural refactor should create minimal compileable skeleton crates for `app-server`, `app-server-protocol`, `app-server-transport`, `app-server-client`, `core`, and `acp` immediately, even if some code moves later. This makes the intended architecture visible and lets Cargo enforce dependency direction early.

Because the current code is not a compatibility target, the refactor does not need a legacy runtime holding crate. Existing runtime code can be deleted or rewritten into the new architecture rather than preserved under a compatibility package.

The scaffold does not need to provide a working MVP or vertical slice. Components should be designed and implemented one by one after their boundaries are grilled and documented.

`openaide-rs/core` owns product modules such as projects, tasks, agents, settings, normalized Chat history, and storage policy. Initial meaningful core modules are `projects`, `agents`, `tasks`, `chat_history`, `settings`, `recovery`, and `storage`. When a workflow crosses multiple product modules, OpenAIDE should add a workflow module named after that workflow, such as `task_creation`, `task_history_sync`, `task_opening`, `agent_identity_migration`, or `startup_recovery`, instead of routing callers through a generic facade or letting app-server coordinate product invariants directly. `openaide-rs/acp` is a pure Agent Client Protocol integration crate for process transport, initialization, session lifecycle, session list/load/resume/delete, updates, permissions, and normalization from ACP payloads. Agent settings, Agent Identity, enabled state, defaults, and product Agent Status interpretation belong in `core`, not `acp`.

`openaide-rs/app-server` should stay a thin orchestration layer. It owns process startup, connected App Shell clients, JSON-RPC lifecycle, transport attachment, shell capability routing, Agent process wiring, and method handlers. Product behavior lives in meaningful `openaide-core` modules and workflow modules rather than being spread across app-server handler modules. OpenAIDE should not create a generic core facade such as `Workbench`, `Core`, `OpenAide`, or `ProductModel` unless a later implementation proves it is a deep interface rather than a pass-through.

Transport and JSON-RPC gateway responsibilities are separate. `openaide-rs/app-server-transport` moves protocol messages over stdio, local sockets, websocket, HTTP/SSE, or native bridges without knowing product method names. `openaide-rs/app-server` owns the JSON-RPC lifecycle: initialize gate, request id matching, method dispatch, error mapping, notifications, and server-initiated requests.

`client_hub` models initialized App Shell client sessions above raw transport connections. It tracks client identity, shell capabilities, pending App Server-initiated requests, subscriptions, connection state, heartbeat or timeout state, and reconnect grace separately from transport connection ids. Initialized client identity is shell-instance identity, not raw transport identity, user identity, or global app identity. A Desktop window or process has one client instance id. A VS Code extension host or window has one client instance id, and recreated webviews reuse it through the extension bridge. A Web App tab has one client instance id stored in browser `sessionStorage` with memory fallback, so reload reconnects but a separate tab is a separate client. A reconnect keeps the same initialized client only when client instance id, local auth, and state root match. Browser reconnects, VS Code webview recreation, Desktop windows, or Mobile view remounts should not force product state to treat every raw transport connection as a new product client unless the App Shell intentionally initializes a distinct client session. `client_hub` is the App Server lifetime authority: AppServer stays alive while at least one initialized client is connected or inside reconnect grace, and graceful shutdown begins only after all initialized clients have disconnected and expired. `client_hub` owns the default 10-second reconnect grace timer and applies transport-specific heartbeat or timeout observations before expiring a client. `client_hub` may process an explicit `client/disconnect` as a fast path, but missing or failed disconnect notifications must behave like ordinary transport loss plus timeout.

`client_hub` stores capability availability and connection state. `capability_broker` executes capability requests by resolving the appropriate client, sending App Server-initiated JSON-RPC requests, and mapping timeout, cancellation, retry, and error outcomes. App Server callers should request capabilities through `capability_broker` rather than addressing shell-specific clients directly.

Task-scoped App Server capability requests are routed to all App Shell clients subscribed to that Task. App Server creates one shared pending request, accepts the first valid response, and then notifies subscribed clients that the request has been resolved so duplicate UI can dismiss. New or reconnected subscribers receive still-pending Task-scoped requests. Each pending request has a stable request id and state such as `pending`, `resolved`, or `interrupted`. Responses are validated against the current pending state. The first valid response atomically moves the request to `resolved`; invalid responses do not resolve the request. App Server emits `serverRequest/resolved` to subscribed clients so duplicate prompts dismiss. Late responses after resolution return a stable stale/resolved protocol error such as `requestAlreadyResolved` and must not affect the already-accepted result. A client that receives a later snapshot after resolution does not see that request as pending. Task-scoped pending requests are owned by App Server, not by any one tab, window, webview, or App Shell client. If one subscribed client disconnects, App Server removes that client from the responder set and keeps the request pending for other subscribed clients. If no clients remain subscribed to the Task but at least one initialized App Shell client remains, the request stays pending and appears when a client subscribes to that Task again. Pending Task-scoped requests are live memory state owned by the AppServer process, not durable actionable state. Task snapshots include pending requests only when they are known to the owning live AppServer. If all initialized clients expire and AppServer shuts down, pending Task-scoped requests are interrupted or detached with the Running Task during shutdown and later recovery; after restart they must not be restored as answerable UI because the Agent may no longer be waiting for the same response. Chat or Task history may show a local event such as interrupted while waiting for permission or user input, but reopening the Task follows normal recovery through `resume`, `load`, or a recoverable unavailable state. Requests tied to the client that initiated an interaction, such as choosing a project folder while creating a Task, are routed only to that originating client and fail if that client disconnects before answering. Global notifications fan out to subscribed or interested clients; global interactive requests require an explicit target client and otherwise fail with a protocol error.

When several clients qualify for a shell-native capability request, `capability_broker` routes deterministically by request scope. Task-scoped user-facing requests go to all subscribed clients and first valid response wins. Client-originated requests go only to the originating client. Secret material requests go only to a client advertising the matching secret-store capability and state root or session ownership; if multiple qualify, App Server prefers the originating client, then the most recently active qualifying client, and otherwise fails with an ambiguity error. File or project picker requests go to the originating client only. Open URL, open file, or reveal commands go to the originating client when triggered by UI, otherwise to the most recently active subscribed client for Task-scoped actions. Global interactive requests with no explicit target fail rather than guessing.

`agent_runtime` in app-server owns live Agent runtime work: launching Agent processes, initializing ACP connections, tracking live Agent connection state, executing ACP session calls such as `session/new`, `session/list`, `session/load`, `session/resume`, and supported `session/delete`, subscribing to ACP updates, normalizing ACP transport failures into app-server observations, and exposing live capabilities observed from initialized Agents. `openaide_core::agents` owns Agent definitions, Agent Identity rules, enabled state, defaults, and status interpretation. App Server wires them together: core Agent modules describe desired Agent state and product rules; `agent_runtime` attempts to realize live connections and reports observations back.

History synchronization is split between app-server orchestration and core reconciliation. App-server `history_sync` owns Agent session-list refresh timing and I/O: startup refresh, manual refresh, reconnect refresh, selecting connected Agents to query, consuming ACP `session_info_update`, and emitting list or state updates after reconciliation. `openaide_core::task_history_sync` owns reconciliation decisions from session-list snapshots: Task projection, local overlay application, Project grouping, and deletion of local normalized Chat history for missing sessions. `history_sync` does not sync arbitrary state, stream Chat, persist Settings, or own file synchronization.

`turn_runner` in app-server owns live Task turn orchestration. It requires the SQLite live Native Session ownership lease before starting Agent work, enforces one active turn per Task across App Server processes, sends the user prompt to the Agent Native Session, consumes the ACP update stream for that turn, asks `capability_broker` for permissions or user input when needed, passes normalized Chat events to `openaide_core::chat_history`, updates `openaide_core::tasks` status, handles explicit `task/cancel`, guards stale responses after cancel, retry, or reconnect, and emits state events through `state_stream`. `turn_runner` uses `agent_runtime` for ACP I/O; `agent_runtime` must not own OpenAIDE turn orchestration.

ACP-to-normalized-Chat conversion lives in `openaide-rs/acp`, not in `turn_runner`. ACP payload semantics belong with the ACP integration crate. `openaide-acp` exposes typed ACP events and normalized OpenAIDE event candidates. `turn_runner` receives normalized events from `agent_runtime` and `openaide-acp`, decides ordering, active-turn applicability, cancellation and stale-response guards, then dispatches accepted events to core modules. `openaide_core::chat_history` stores normalized Chat events and builds renderable Chat state.

Accepted normalized Chat events are persisted before they are emitted to `state_stream`. `turn_runner` accepts a normalized event only after stale guards pass. If one ACP update maps to multiple normalized Chat events, `openaide_core::chat_history` persists those events atomically together. App Server emits the corresponding Task events or snapshot update only after persistence succeeds. If persistence fails or storage becomes unavailable while a turn is running, App Server stops accepting new Agent updates for rendering, stops the turn locally when it can persist that state, surfaces renderable storage error state to connected clients, and does not present non-durable Agent output as if it will survive reload. App Server should close or detach the Agent transport as best effort. If storage cannot persist the interrupted or storage-error Task state, App Server must not cleanly release live ownership; the ownership lease expires and later recovery treats the takeover as unclean detach.

Normalized Chat history persistence lives in `openaide_core::chat_history`. App-server streams normalized Chat events into the core Chat history module and asks core Task-opening or snapshot modules for renderable Chat state. Core modules own storage policy, schema versioning, cleanup when sessions disappear, and the invariant that a reloaded Chat looks the same when the local projection is valid. Local normalized Chat is a projection of Agent-owned session state, not the source of truth for what the Agent will see. Chat history records must carry enough validity metadata, such as Agent Identity, Native Session id, schema version, live ownership epoch, and any Agent-provided revision when available, so App Server can reject or rebuild stale projections before live interaction. App-server must not know the normalized Chat file layout.

`openaide_core::chat_history` stores normalized Chat events as the durable source and maintains a materialized render state for fast Task snapshots. Snapshot reads use the materialized render state rather than replaying the full event log every time. On startup, migration, or corruption recovery, the render state can be rebuilt from normalized events. Tests must verify that rebuild produces the same user-visible Chat render state.

`state_stream` owns subscriptions, typed scopes, snapshot responses, product event fanout, unsubscribe cleanup, and reconnect behavior. It uses `client_hub` to address connected clients and uses meaningful `openaide-core` modules, such as `tasks`, `agents`, `projects`, `settings`, and `task_opening`, to build App Server-owned state projections. `client_hub` answers who is connected and what capabilities are available; `state_stream` answers which clients are watching which product state and what they should receive next.

Live Chat presentation has two deliberately separate states. App Server owns authoritative received text and persists each accepted chunk before emitting `chatItemAppended` or `chatItemChunk`; the first event establishes stable message identity and later chunks carry ordered append data, including explicit finalization. Frontend may keep disposable per-message presentation state such as visible grapheme count, a short activity barrier, or follow-scroll state. Presentation state must not modify persisted text, delay permission handling, replay on history or resync, or become part of App Server protocol. A terminal Task transition finalizes any persisted streaming messages even when cancellation wins the race against the Agent response.

App-server protocol handlers are grouped by protocol resource, such as `task`, `agent`, `project`, `settings`, and `capability`. Handlers stay thin: they validate protocol params, call meaningful core modules or app-server orchestration modules such as `agent_runtime`, `history_sync`, and `capability_broker`, and return protocol responses. Product decisions do not live in handlers.

The app-server composition root type is `AppServer`, not `runtime`. The term runtime is overloaded by the old architecture and should not be used as the main server/module name in the refactor.

One AppServer process owns its initialized App Shell clients, live Agent processes, subscriptions, pending capability requests, and any Native Sessions whose live ownership locks it currently holds. The shared state root can be opened by more than one AppServer process, but shared persisted state is updated only through App Server storage modules and storage-level concurrency controls. Client identity, capabilities, connection state, and pending App Server-initiated requests stay in `client_hub`; product state subscriptions stay in `state_stream`.

AppServer supports multiple initialized App Shell clients at the protocol level. Desktop, Web App, Mobile App, and VS Code Extension may attach concurrently to the same local AppServer and shared state root. They may also run through separate AppServer processes over the same state root for persisted history and settings. The invariant is one live owner per Native Session, not one AppServer process per state root.

Running Tasks are owned by the App Server process that holds live ownership for the Task's Native Session, not by the App Shell client that started them. App Server blocks a second live interaction for the same Native Session across clients and App Server processes. Clients attached to the owning AppServer may observe the Running Task live and answer Task-scoped capability requests. Other AppServers may read persisted Task history and list state, but must not call `session/load`, `session/resume`, send, cancel, consume live Agent updates, or append normalized Chat for that Native Session until ownership is released or expires. The originating client is recorded only for client-scoped capability requests.

If the last client unsubscribes from a Running Task while AppServer still has at least one initialized App Shell client, App Server keeps the Agent Native Session running. Loss of Task subscribers is not cancellation, unload, or ownership transfer. App Server continues receiving ACP updates, persists normalized Chat, updates Task state, and sends no Task UI events until a client subscribes again. A later subscriber receives a fresh Task snapshot with current Chat, status, and pending Task-scoped requests. This rule does not make AppServer durable after all App Shell clients close; zero initialized clients triggers graceful shutdown after the reconnect grace period. Cancelling a Running Task requires explicit user intent through a protocol method such as `task/cancel`; idle cleanup is outside v1 and must not be tied to viewer count.

The App Server event model is snapshot plus events. When a client subscribes to or opens a scope, App Server sends a full snapshot for that scope and then ordered events for subsequent changes observed by that AppServer. Task events are ordered per Task inside the owning AppServer. Global list, settings, and Agent events are ordered per AppServer process. Reconnect in v1 does not require durable event replay; clients resubscribe and receive fresh snapshots. Cross-process freshness is handled by storage reads, startup/manual refresh, and later optional storage notifications rather than a durable event bus in v1. Task snapshots include unresolved Task-scoped capability requests known to the owning AppServer so browser reloads, Desktop windows, Mobile remounts, and recreated VS Code webviews attached to that server can render pending prompts. Meaningful `openaide-core` modules own the state projections used to build snapshots and apply product decisions; app-server owns subscription tracking and event delivery.

Subscriptions use one generic App Server Protocol method with typed scopes rather than one subscribe method per resource. `state/subscribe` accepts a product scope such as `task`, `taskNavigation`, `agents`, `projects`, or `settings`, and returns the initial snapshot plus cursor for that scope. `state/unsubscribe` removes the subscription and is idempotent for already-removed scopes. Resource-specific command methods remain resource-specific, such as `task/create`, `task/send`, `agent/update`, and `settings/update`; only subscription mechanics are centralized behind `state/subscribe`.

After a subscription snapshot, App Server sends ordered App Server events with an explicit scope, `previousCursor`, `cursor`, and typed payload enum rather than JSON Patch or generic record deltas. The event wrapper is the transport-neutral App Server Protocol event envelope; payload variants remain product-specific, such as Task snapshots, Task navigation updates, Chat item appends, Chat chunks, and pending request updates. `previousCursor` lets Frontend prove event continuity against its current cursor before applying the event. If App Server cannot cheaply express a change as a typed event payload, it sends a typed invalidation or snapshot replacement payload and the client refreshes the affected scope through `state/subscribe` or a read method. OpenAIDE does not use JSON Patch in the App Server Protocol.

OpenAIDE v1 does not support per-method notification opt-out. Clients receive notifications for the scopes they subscribe to, and App Server routes typed notifications by subscription state. Notification suppression can be added later only if a real client cannot handle a notification family.

The App Server Protocol is a single evolving protocol for now. OpenAIDE does not use explicit method namespaces such as `v1/task/create` or `v2/task/create`, and it does not need a heavy version-negotiation system before stable external client support exists. App-server and App Shells are expected to ship together during this refactor phase, and breaking protocol changes are allowed. `initialize` may exchange app and capability information, but explicit compatibility/versioning should be added only when OpenAIDE publishes stable external client support or generated compatibility artifacts need it.

`initialize` is small and mandatory. It identifies the App Shell kind and shell-instance client through fields such as `shellKind`, `clientInstanceId`, and optional `shellInstanceLabel`, provides client capabilities such as native file picker, secret storage, shell open-url or open-file helpers, workspace or project hints, and optional initial shell context such as the current VS Code workspace path. App Server returns server capabilities, startup status, `serverLifecycle`, and redacted product or environment facts useful for UI boot. Successful `initialize` returns `serverLifecycle: "running"`; initialization failure uses a structured protocol error rather than a partially initialized state. `rpc_gateway` rejects non-`initialize` requests before successful initialization. `initialize` must not load Task history, subscribe to state, start user work, or perform heavy version negotiation. Normal App Server startup may already be connecting enabled Agents, but `initialize` itself does not create Tasks or start Agent turns.

Client capabilities can change after `initialize`. The App Server Protocol provides `client/capabilitiesChanged` for App Shells to report changes such as VS Code workspace changes, secret storage availability, native file picker bridge reconnects, browser local-helper loss, or Desktop native permission changes. App Server updates `client_hub`, recomputes affected capability state, emits typed capability or status notifications, and fails, reroutes, or keeps pending capability requests according to `capability_broker` rules. Frontend renders capability state from App Server snapshots and events rather than checking App Shell names.

App Server exposes errors through structured protocol errors and renderable state. Request failures return JSON-RPC errors with stable `code`, user-facing `message`, and structured `data`. State failures that belong to already-rendered UI, such as Agent unavailable, Chat history unavailable, Task storage error, or interrupted recovery, are represented in snapshots and events as renderable error state. Frontend must not parse English messages to make decisions. Every recoverable error has a stable kind. App Server maps low-level ACP, storage, and transport errors into product error kinds before sending them to clients. Sensitive paths, environment values, secrets, and setup details are redacted from protocol data by default; diagnostic detail can remain in local logs or support exports subject to redaction policy.

Support Export is metadata-only for external side-effect diagnostics in v1. It may include raw OpenAIDE-generated opaque identifiers such as task id, turn id, export-local diagnostic id, and lease epoch metadata. User-provided, path-derived, provider-derived, shell-provided, and Agent-provided identifiers are tokenized or redacted unless an adapter proves they are opaque and non-sensitive; Native Session id is therefore tokenized or redacted by default. It may include error kinds, timestamps, storage and transport diagnostics, live ownership owner and epoch metadata, redacted or truncated `embeddedSnapshot` content fingerprints for same-bytes correlation, and redacted ACP or normalized-event summaries. Local filesystem paths are tokenized by default with stable per-export tokens such as `path:1` and `path:2`; the same original path maps to the same token within one export, but tokens are not stable across exports unless a later explicit opt-in mode requires it. Basenames may be included only when already user-visible and safe; otherwise they are redacted. Support Export must not automatically include file contents, full content hashes, full project diffs, full terminal output, secrets, environment dumps, raw ACP envelopes, private Agent metadata, endpoint records, auth tokens, usernames, home directories, private domains, temporary preview hosts, or machine-specific setup details. It may include redacted counts or statuses such as file edit observed, terminal command event observed, permission request interrupted, or storage failure during turn when those facts are already represented in normalized redacted Chat or diagnostics. Users can manually attach project diffs or terminal output to a bug report later when they choose to share that content. A future advanced export mode may include raw paths only after an explicit warning and user choice.

Support Export may include sent attachment label snapshots only when they are the same safe labels already visible in Chat and remain under normal export redaction rules. It must not add raw paths, resolver ids, raw root metadata, hidden disambiguators, shell picker details, or other attachment-only local facts simply because an attachment existed.

Redacted or truncated `embeddedSnapshot` fingerprint tokens are stable only within one Support Export by default. That is enough to correlate repeated references to the same sent bytes inside a single diagnostic bundle. Cross-export-stable content fingerprints would become durable content identifiers, so they require a future explicit advanced export mode with a user warning.

Local diagnostic logs are an on-machine debugging aid, not a sharing format. They are scoped per state root, rotated by default after 7 days or when a configured size cap is reached, and can retain richer local detail than Support Export only while kept on the user's machine. Local logs may include sent attachment safe label snapshots, attachment kind, sent mode, delivery status, and redacted or truncated `embeddedSnapshot` fingerprints when useful for debugging. They must not include raw paths, resolver ids, file URIs, shell picker details, hidden root metadata, embedded file contents, secrets, auth tokens, raw environment dumps, raw request payloads, or full `embeddedSnapshot` content hashes unless a specific temporary targeted diagnostic path is explicitly designed. Full embedded content hashes remain internal to storage/runtime paths that need idempotency, dedupe, or in-process debugging, because full hashes in logs become long-lived content identifiers. Any user-shared diagnostic bundle must be generated through Support Export so the stricter redaction and tokenization policy is applied.

OpenAIDE does not send automatic telemetry, crash uploads, or background diagnostic network calls in v1. Crash details and diagnostic logs remain local unless the user explicitly creates a Support Export or manually shares files. Future opt-in telemetry or crash reporting would be a separate product feature, not a default behavior hidden inside the App Server, App Shells, or Frontend.

Error kinds exist on both sides of the app-server seam with different roles. `openaide-core` defines internal product error enums where product decisions happen. `openaide-app-server-protocol` defines stable wire error kinds and JSON-RPC codes that clients can rely on. App-server maps internal errors to protocol errors at handler and state boundaries. Wire kinds should be fewer and more stable than internal error variants. Raw `rusqlite`, ACP transport, OS, and Agent-specific errors must not leak over the protocol. For example, an internal `TaskOpeningError::LocalChatSchemaTooNew` maps to a wire `chatHistoryUnavailable` kind, an internal `AgentLaunchError::ExecutableMissing` maps to `agentSetupRequired`, and a late response to a resolved App Server request maps to `requestAlreadyResolved`.

`openaide-app-server-protocol` defines UI-ready product snapshots, not only low-level records or deltas. Snapshot types include `TaskSnapshot`, `TaskListSnapshot`, `AgentListSnapshot`, `ProjectListSnapshot`, `SettingsSnapshot`, renderable Chat model, pending requests model, and renderable error state. These types carry App Server-owned product decisions so Frontend does not reconstruct workflow state independently. Protocol snapshots must not include CSS, layout, component names, shell placement, or local React state shape.

The App Server Protocol follows Codex app-server conventions where practical: bidirectional JSON-RPC-style requests, responses, and notifications; resource methods shaped as singular `<resource>/<method>`; request payloads named `*Params`; responses named `*Response`; notifications named `*Notification`; camelCase wire fields; and generated TypeScript protocol artifacts from the Rust protocol crate. JSON-RPC is the wire shape across pluggable transports such as stdio, local socket, websocket, browser bridge, or native IPC. `openaide-rs/app-server-protocol` is the protocol source of truth. Rust and TypeScript both have protocol types, but TypeScript mirrors the App Server Protocol through generation or explicit compatibility checks rather than defining an independent contract. The protocol defines message types, shared records, and a minimal client-facing API that Frontend calls to subscribe to product state and events, send user intents, answer App Server requests, and read shell capabilities. App Server can initiate user-facing requests through the App Server Protocol, including permissions, file choices, confirmation prompts, auth/setup requests, terminal/file events, and progress updates. Frontend renders these requests, while App Shells fulfill shell-native capability parts when required. Frontend depends on that API rather than on a transport implementation. Until generation exists, manually maintained TypeScript must mirror the intended `openaide-rs/app-server-protocol` shape and tests must cover the external protocol shape so protocol drift is caught at the boundary rather than hidden by mocks.

Shell capabilities are modeled as observable App Server Protocol state. Most capabilities are stable after startup, but they can change when workspaces, local bridges, permissions, native integrations, or external binaries appear or disappear. Shared Frontend should render from capability state rather than branching on App Shell names.

Considered alternatives were keeping VS Code as the primary architecture, treating Host as the cross-platform product term, letting Frontend own workflow state, creating separate protocols for each App Shell, and generating all TypeScript protocol bindings before the Web App exists. The chosen path prioritizes shared product behavior, reusable Frontend rendering, and independent App Shell delivery while avoiding premature schema-generation infrastructure.
