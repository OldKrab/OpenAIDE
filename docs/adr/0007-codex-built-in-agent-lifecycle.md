# Built-In Agent Lifecycle

OpenAIDE is ACP-only for the current product direction, and Codex and OpenCode are built-in Agents. Agent definitions contain identity and launch information only; auth methods come from ACP `initialize.authMethods`, and Configuration Options come from session setup and update messages.

For Codex, OpenAIDE tries `codex-acp` from `PATH`, then automatically falls back to `npx -y @agentclientprotocol/codex-acp@<exact-version>`. Each OpenAIDE release owns and tests that exact fallback version; upgrading it requires changing the pin in an OpenAIDE release. OpenCode retains its existing `opencode acp` / `npx -y opencode-ai acp` launch policy until a tested OpenCode pin is adopted separately. The `npx` fallback is an Agent launch path even when it downloads the package; it does not create a separately detectable package-installed state and does not require a setup confirmation. A missing launch prerequisite such as `npx` produces setup-required Agent Status, while a fallback that starts but cannot download, launch, or initialize the Agent produces failed Agent Status with explicit retry. Because failure to launch `npx` does not prove whether Node.js is absent or merely unavailable to the App Shell environment, the regular-user setup surface reports that Node.js tools are unavailable and offers installation, recheck, and setup-help recovery without exposing `npx`, `PATH`, or raw command errors. Opening the installer does not trigger background or focus-based detection: the user explicitly chooses Check again, and a failed recheck may then offer Reload VS Code and continue. The Agent is selectable for new Tasks after successful ACP initialization and while it has no known auth/setup blocker; a real Prepared Task or Task operation may subsequently reveal `auth_required` and pause work for explicit authentication. Task-specific Configuration Options are discovered from each Task's Native Session after `task/create`; if Task session setup fails, OpenAIDE shows renderable Task preparation error state instead of falling back to a mock.

App Server owns the structured setup reason and retry result. Shared Frontend owns the product explanation and action layout, while each App Shell advertises and performs only the external-link, reload, or equivalent recovery actions it supports; shared product state does not hard-code VS Code commands.

OpenAIDE does not guess Configuration Option defaults. A controlled no-options state is allowed only when the Agent successfully reports no Configuration Options or the Task session exposes an explicitly recoverable no-options state.

After a user completes Agent auth or setup in settings, OpenAIDE attempts to reconnect automatically. Agent settings also exposes an explicit retry action so users can rerun connection/setup checks when auto reconnect fails or external state changes.

Built-in Agents are fixed product entries, not editable templates. Each built-in exposes enable/disable, setup/auth, probe, retry, and status actions; users who need a different command, arguments, or environment create a Custom Agent instead.

The first production transport is ACP over stdio because current built-ins use process stdio. Custom Agents in the first iteration also use stdio process transport. Other ACP transports, including HTTP/WebSocket draft transport, remain tracked planned support and do not block built-in ACP Agent support.
